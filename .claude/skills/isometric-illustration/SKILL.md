# Isometric Illustration Skill

> 키워드/컨셉을 받아 Linear 스타일의 아이소메트릭 SVG 일러스트를 생성하는 워크플로우.
> "무엇을 그릴까"가 아니라 **"이 개념을 어떤 배치 패턴으로 번역할까"**가 핵심이다.

## 활성화 조건

| 의도 | 트리거 예시 |
|------|-----------|
| 일러스트 생성 | "일러스트 만들어줘", "SVG 그려줘", "비주얼 만들어줘" |
| 개념 시각화 | "이 개념을 시각화해줘", "아이소메트릭으로 표현해줘" |
| 레퍼런스 스타일 | "Linear 스타일로", "레퍼런스처럼" |

---

## 파이프라인 총론

```
키워드 → 메타포 추출 → 컨셉화 → 구조화 → 시각 마무리
```

모든 단계에서 "새 형태를 발명"하지 않는다.
**파라미터를 바꿔 의미를 바꾸는 것**이 이 시스템의 본질이다.

---

## 1단계: 키워드 → 시각적 메타포 추출

사용자의 키워드/컨셉에서 **물리적 배치 패턴**을 추출한다.

### 메타포 사전

키워드가 아래 패턴 중 어디에 해당하는지 판단한다:

| 패턴 | 배치 방식 | 대표 키워드 |
|------|----------|-----------|
| **기반/원칙** | 수평 레이어 적층 | 기반, 원칙, 토대, 신뢰, 안정, 깊이 |
| **모듈/구성** | 독립 블록 클러스터 | 구조, 모듈, 에이전트, 분업, 조합, 아키텍처 |
| **속도/모멘텀** | 대각선 캐스케이드 | 속도, 성장, 가속, 진행, 흐름, 처리량 |
| **계층/위계** | 크기 차이 피라미드 | 위계, 단계, 상위-하위, 계층, 종속 |
| **확장/분화** | 중심에서 방사 | 확장, 파생, 생태계, 네트워크, 분기 |
| **수렴/통합** | 외곽에서 중심으로 | 통합, 수렴, 집중, 합류, 단일화 |
| **순환/반복** | 링형 또는 순환 배치 | 사이클, 반복, 피드백, 루프, 리듬 |
| **대비/전환** | 좌우 또는 전후 분할 | Before/After, 비교, 전환, 반전 |

### 프로세스

1. 사용자 키워드를 받는다
2. 가장 가까운 패턴 1~2개를 제안한다
3. 사용자에게 확인한다: "이 개념을 [패턴명]으로 표현하려 합니다"

**예시:**
- "확장성" → **확장/분화** (중심에서 바깥으로 모듈이 뻗어나가는 배치)
- "디자인 시스템" → **기반/원칙** (수평 레이어로 쌓인 구조적 토대)
- "워크플로우 자동화" → **속도/모멘텀** (대각선으로 흘러가는 처리 카드)

## 2단계: 메타포 → 컨셉화

선택된 패턴을 구체적인 **프리미티브 + 배치 규칙**으로 변환한다.

### 2.1 프리미티브 선택

아이소메트릭 일러스트의 기본 도형은 단 2종:

| 프리미티브 | 용도 | 구성 |
|-----------|------|------|
| **Prism** (육각 프리즘) | 컨테이너, 모듈, 플랫폼 | 상단면 + 좌측면 + 우측면, 상단 V자 분할선 |
| **Card** (평행사변형) | 문서, 작업, 데이터 단위 | 상단면 + 우측면만, 납작한 프리즘 |

**규칙: 한 일러스트에 프리미티브 1종만 사용한다.**
의미는 배치 패턴으로 만든다. 도형 종류를 섞지 않는다.

### 2.2 배치 규칙 정의

패턴별로 결정해야 할 파라미터:

```
{
  primitive: 'prism' | 'card',
  count: 3~15,               // 인스턴스 수
  heroIndex: number,          // 하이라이트 요소 위치 (1개만)
  arrangement: {
    axis: 'horizontal' | 'diagonal' | 'radial' | 'cluster',
    offset: { x, y },        // 인스턴스 간 이동량
    heightCurve: 'uniform' | 'pyramid' | 'asymmetric-wave' | 'descending',
  },
  hierarchy: {
    activeCount: number,      // D0D6E0 스트로크 적용 수
    subtleCount: number,      // 3E3E44 스트로크 적용 수
  }
}
```

### 2.3 컨셉 카드 작성

사용자에게 보여줄 한 줄 컨셉:

```
[컨셉]
키워드: 확장성
메타포: 확장/분화
프리미티브: Prism (육각 프리즘)
배치: 중앙 1개 대형 + 주변 4~6개 소형이 방사형으로 배치
히어로: 중앙 프리즘 (가장 크고 밝은 스트로크)
높이 곡선: 중앙→외곽으로 점진적 감소
인스턴스 수: 7개
```

**승인 없이 3단계 진행 금지.**

## 3단계: 컨셉 → 구조화

승인된 컨셉을 SVG 좌표계로 변환한다.

### 3.1 캔버스 규격

| 파라미터 | 값 | 근거 |
|---------|---|------|
| viewBox | ~270 x ~270 (±15%) | 레퍼런스 3종 평균. 정사각에 가깝게 |
| fill | `none` (캔버스 자체) | 다크 배경은 부모 컨테이너가 담당 |

### 3.2 아이소메트릭 각도

모든 경사면은 동일한 각도를 사용한다:
- 각도: ~26.57° (atan(1/2)) — 표준 다이메트릭 프로젝션
- SVG path에서는 이 각도가 `a` (arc) 커맨드 또는 직선 좌표로 표현됨
- **모든 인스턴스가 동일한 각도를 공유해야 한다**

### 3.3 크기 결정

```
히어로 높이 = viewBox 높이의 40~50%
최소 높이 = 히어로 높이의 5~8%
중간 단계 = heightCurve에 따라 보간
```

### 3.4 Z-order

- 뒤(먼 곳)부터 앞(가까운 곳) 순서로 SVG에 배치
- 뒤쪽 요소가 먼저 렌더링되어 앞쪽에 가려짐

### 3.5 상단면 V자 선

모든 Prism/Card에 상단면 내부 분할선을 넣는다:
- 좌측 아래에서 중앙 위로, 다시 우측 아래로 — V자 패턴
- stroke: `--vdl-800` (가장 어두운 구조선)

## 4단계: 구조 → 시각 마무리

`resources/visual-grammar.md` Read → 토큰과 규칙을 적용한다.

### 4.1 적용 순서

1. **Fill 적용**: 모든 면에 `--vdl-950` (배경 동화)
2. **스트로크 위계 적용**: 히어로 → active, 나머지 → default/subtle
3. **내부선 적용**: 상단면 V자, 컨텐츠 라인(히어로만)
4. **드롭 쉐도우 적용**: 밀도에 비례한 stdDeviation
5. **여백 확인**: 캔버스 가장자리에 최소 8px 패딩
6. **filter ID 유니크**: 일러스트별 prefix 부여 (`fig{N}_f{n}`)

### 4.2 히어로 요소 디테일

히어로 카드/프리즘 내부에만 컨텐츠를 암시하는 라인을 추가할 수 있다:
- 1~2줄의 짧은 수평선 (텍스트 암시)
- stroke: `--vdl-700`
- 히어로 이외의 요소는 비어있어야 한다

### 4.3 최종 검증

| # | 체크 |
|---|------|
| 1 | 프리미티브 1종만 사용했는가? |
| 2 | 유채색 0인가? |
| 3 | 히어로 1개만 `--vdl-200` 스트로크인가? |
| 4 | 모든 fill이 `--vdl-950`인가? |
| 5 | strokeWidth가 `0.5` 통일인가? |
| 6 | strokeLinecap이 `round`인가? |
| 7 | 아이소메트릭 각도가 전체 통일인가? |
| 8 | 캔버스가 거의 정사각(±15%)인가? |
| 9 | filter ID가 고유한가? |
| 10 | 배치 패턴만으로 메타포가 읽히는가? |

## 핵심 원칙

- **형태가 아니라 배치가 의미를 만든다** — 새 도형을 발명하지 말고 배치 패턴을 바꿔라
- **컨셉 카드 승인 없이 SVG 작성 금지** — 2단계에서 반드시 사용자 확인
- **파라미터로 재현 가능해야 한다** — 감이 아니라 규칙으로 결정한 값만 사용
- **히어로는 정확히 1개** — 시선 앵커가 2개 이상이면 위계가 무너진다
- **빈 공간은 채우지 않는다** — 다크 배경의 여백이 형태만큼 중요하다
