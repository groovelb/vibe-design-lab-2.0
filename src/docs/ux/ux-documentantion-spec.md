# UX 기반 컨텍스트 엔지니어링: 5개 문서 충족 요건

> 이 문서들의 순서는 의사결정의 순서입니다.
> 위 레이어가 결정되지 않으면 아래 레이어를 시작하지 않습니다.
> AI에게 시키는 것이 아니라, 인간이 먼저 판단 기준을 확보하는 과정입니다.

---

## 1. project-summary

**파일명:** `docs/ux/project-summary.md`

### 이 문서가 하는 일
서비스가 왜 존재하는지, 무엇을 해결하는지 정의한다.
이후 모든 판단의 기준점이 된다.

### 여기서 결정하는 것
- 이 서비스가 해결하는 문제 (사용자 관점)
- 핵심 가치 — 기능이 아닌 경험적 약속
- 제공할 것과 제공하지 않을 것 (Non-goal 명시)
- 성공이 어떤 상태인지 (정성적으로도 괜찮음)
- 타겟 사용자가 누구인지 (퍼소나 수준까지는 아니어도)

### 다음 단계에 넘겨주는 것
ux-flow가 화면을 설계할 때 "이게 핵심 가치에 맞는가"를 판단하는 기준.
technical이 기술 선택을 할 때 "이 복잡도가 서비스 목표에 맞는가"를 판단하는 기준.

### AI가 이걸 어떻게 쓰는지
이 문서가 있으면 AI가 "이 기능 추가할까요?"라고 묻지 않는다.
대신 "project-summary의 핵심 가치 기준으로 이 기능은 범위 밖입니다"라고 스스로 판단한다.
없으면 AI는 그냥 시키는 대로 다 만든다 — 방향 없는 기능 덩어리가 된다.

### 이 문서가 완성됐다는 신호
- "이 서비스는 ___을 위해 존재한다"를 한 문장으로 말할 수 있다
- "이건 우리가 안 하는 거야"를 명확히 말할 수 있는 항목이 최소 2개 있다
- 성공 기준을 읽었을 때 팀원이 다른 해석을 하지 않는다

## 2. ux-flow

**파일명:** `docs/ux/ux-flow.md`

### 이 문서가 하는 일
project-summary의 "무엇을"을 "어떻게 경험시킬 것인가"로 변환한다.
사용자가 서비스 안에서 움직이는 경로와 구조를 설계한다.

### 여기서 결정하는 것
- 정보 구조 (IA) — 콘텐츠가 어떻게 분류되고 연결되는지
- 유저 플로우 — 주요 태스크를 완료하는 경로
- 화면 목록 — 존재해야 하는 화면과 각 화면의 목적
- 인터랙션 원칙 — 이 서비스에서 일관되게 적용할 UX 규칙
- 필요한 데이터 모델의 윤곽 (technical에서 확정하지만, ux-flow가 먼저 요구사항을 정의)
- 필요한 외부 연동의 윤곽 (지도 API가 필요한지, 소셜 로그인이 필요한지 수준)

### 다음 단계에 넘겨주는 것
design-system-optimization에게: 어떤 컴포넌트 패턴이 반복되는지, 어떤 인터랙션이 있는지.
technical에게: 어떤 데이터가 필요한지, 어떤 상태를 관리해야 하는지.
contents에게: 어떤 화면에 어떤 종류의 텍스트와 미디어가 들어가는지.

### AI가 이걸 어떻게 쓰는지
이 문서가 있으면 AI가 화면 구조를 임의로 만들지 않는다.
"ux-flow 기준으로 이 화면은 3단계 이후에 등장합니다"처럼 구조를 지킨다.
없으면 AI는 자신이 본 비슷한 서비스의 패턴을 그대로 복사한다 — project-summary의 의도와 무관하게.

### 이 문서가 완성됐다는 신호
- 유저 플로우를 따라가면 시작부터 핵심 태스크 완료까지 막히는 지점이 없다
- 화면 목록을 보면 project-summary의 핵심 가치가 어디서 경험되는지 보인다
- "이 화면은 왜 있어요?"라는 질문에 모두 답할 수 있다

## 3. design-system-optimization

**파일명:** `docs/ux/design-system-optimization.md`

### 이 문서가 하는 일
기존 디자인 시스템 위에 이 프로젝트만의 시각 언어를 올린다.
어떻게 보여야 하는지(visual direction)와, 그걸 어떤 토큰과 컴포넌트로 구현할지를 결정한다.

### 여기서 결정하는 것
- Visual direction — 색, 타이포그래피, 공간감, 무드의 방향성 (레퍼런스 포함)
- Semantic token 커스텀 — 기존 디자인 시스템 기본값에서 변경할 토큰만 (delta 관리)
  - 예: `color.primary`, `spacing.base`, `typography.heading` 등
- 필요한 컴포넌트 리스팅 — 기존 디자인 시스템에 없거나 변형이 필요한 것만
  - 있는 것: 기존 컴포넌트명 참조
  - 없는 것: 신규 제작 목록
  - 변형이 필요한 것: 어떻게 달라야 하는지

### 다음 단계에 넘겨주는 것
technical에게: 토큰 구조가 확정되어야 theme 파일 작성이 가능함.
contents에게: 이미지/미디어가 들어가는 컴포넌트의 사이즈 기준과 비율.

### AI가 이걸 어떻게 쓰는지
이 문서가 있으면 AI가 기존 디자인 시스템의 기본 스타일을 무분별하게 쓰지 않는다.
정의된 토큰 기준으로 일관된 UI를 생성하고, 매번 스타일을 지시할 필요가 없다.
없으면 AI는 프롬프트마다 다른 스타일을 쓰고, 컴포넌트마다 결과물이 제각각이 된다.

### 이 문서가 완성됐다는 신호
- Visual direction 레퍼런스를 보면 팀원이 "이런 느낌이구나"를 바로 이해한다
- 변경할 토큰 목록이 있고, 변경하지 않을 것은 명시적으로 "기존 시스템 기본값 유지"라고 적혀있다
- 컴포넌트 리스트를 보면 ux-flow의 화면 목록을 커버할 수 있는지 확인된다

## 4. technical

**파일명:** `docs/ux/technical.md`

### 이 문서가 하는 일
ux-flow가 요구한 경험을 실제로 구현하기 위한 기술적 판단을 내린다.
무엇을 만들 수 있는지의 경계를 설정한다.

### 여기서 결정하는 것
- 데이터 모델 — 어떤 엔티티가 있고 어떤 관계인지
- API 구조 — 어떤 엔드포인트가 필요한지 (REST/GraphQL 수준 결정 포함)
- 외부 라이브러리 선택과 그 이유 — "왜 이걸 썼는지"가 중요
- 기술 제약 — 이 프로젝트에서 하지 않기로 한 기술적 선택과 이유
- 상태 관리 전략 — 어떤 데이터를 어느 수준에서 관리할지

### 다음 단계에 넘겨주는 것
contents에게: 이미지, 영상, 텍스트가 어떤 엔티티에 속하는지.
그에 따라 contents의 관리 정책이 결정된다.

### AI가 이걸 어떻게 쓰는지
이 문서가 있으면 AI가 컴포넌트를 만들 때 임의로 상태 관리 방식을 선택하지 않는다.
"이 프로젝트는 Zustand를 쓰고, 서버 상태는 React Query로 분리한다"는 기준이 있으면
AI는 그 기준을 지키며 구현한다.
없으면 AI는 세션마다 다른 패턴을 쓰고, 코드베이스가 일관성 없이 쌓인다.

### 이 문서가 완성됐다는 신호
- 데이터 모델을 보면 ux-flow의 화면에서 필요한 데이터가 모두 설명된다
- 외부 라이브러리 선택에 "왜"가 명시되어 있다
- "이 기능은 기술적으로 어떻게 동작해요?"라는 질문에 이 문서로 답할 수 있다

## 5. contents

**파일명:** `docs/ux/contents.md`

### 이 문서가 하는 일
서비스 안의 모든 텍스트, 이미지, 영상, 데이터 연동 콘텐츠가
어떤 규칙으로 존재하고 관리되는지를 정의한다.

### 여기서 결정하는 것
- 상수 텍스트 — 버튼 레이블, 에러 메시지, 빈 상태 문구 등 코드에 고정되는 텍스트
- 메시지 정책 — 어떤 상황에 어떤 톤으로 말할지 (성공/실패/경고/안내)
- 이미지·영상 관리 규칙 — 사이즈, 포맷, 비율, 대체 텍스트 정책
- 데이터 연동 콘텐츠 — CMS나 API에서 오는 콘텐츠의 구조와 표시 규칙
- 다국어 대응 여부와 방식 (필요한 경우)

### 다음 단계에 넘겨주는 것
이 문서는 마지막 레이어다. 위 4개 문서의 결정을 받아서 구체화한다.
AI 코딩 세션에서 직접 참조되는 최종 구현 가이드가 된다.

### AI가 이걸 어떻게 쓰는지
이 문서가 있으면 AI가 임의로 에러 메시지를 작성하지 않는다.
"데이터를 불러오지 못했습니다" 대신 정의된 메시지 정책을 따른다.
이미지 컴포넌트를 만들 때도 정의된 비율과 대체 텍스트 규칙을 적용한다.
없으면 AI는 자신이 적절하다고 생각하는 텍스트를 생성하고, 그게 서비스 톤과 맞지 않아도 알 수 없다.

### 이 문서가 완성됐다는 신호
- 주요 화면의 모든 상수 텍스트가 정리되어 있다
- 에러가 났을 때, 성공했을 때, 빈 화면일 때 각각 어떻게 말할지 예시가 있다
- 이미지를 넣을 때 개발자가 이 문서만 보고 판단할 수 있다

## 문서 흐름 요약

```
project-summary (왜, 무엇을)
  ↓ 경험으로 변환
ux-flow (어떻게 경험시킬지)
  ↓ 시각 언어와 컴포넌트로 구체화
design-system-optimization (어떻게 보여줄지)
  ↓ 구현 가능한 기술 구조로 확정
technical (어떻게 만들지)
  ↓ 실제 콘텐츠와 운영 규칙으로 마무리
contents (무엇을 넣고 어떻게 관리할지)
```

각 레이어는 위 레이어의 결정을 입력으로 받아서, 자기 레이어의 질문에만 답한다.
이 순서를 지키는 것 자체가 컨텍스트 엔지니어링이다.

## Claude Code 적용 구조

### 디렉토리 구조

```
project-root/
├── CLAUDE.md                              ← 항상 로딩. 50줄 이하
├── docs/
│   └── ux/                                ← 실제 내용이 있는 원본 문서
│       ├── project-summary.md
│       ├── ux-flow.md
│       ├── design-system-optimization.md
│       ├── technical.md
│       └── contents.md
└── .claude/
    └── rules/
        └── ux-architecture.md             ← 자동 로딩. 지도 역할만
```

`docs/ux/`는 사람이 읽고 관리하는 원본이다.
`.claude/rules/ux-architecture.md`는 Claude가 매 세션 자동으로 로딩하는 지도다.
지도는 짧게 유지하고, 실제 내용은 필요할 때 docs/ux/에서 읽어온다.

---

### ux-architecture.md — rules에 등록되는 유일한 UX 문서

자동 로딩되므로 짧게 유지한다. 판단 기준과 참조 경로만 담는다.

```markdown
# UX Architecture

## 핵심 원칙
- 문서에 정의되지 않은 판단이 필요하면 구현 전에 먼저 질문한다
- 컴포넌트 신규 제작 전 design-system-optimization 문서의 컴포넌트 리스트를 먼저 확인한다
- 상수 텍스트는 임의로 작성하지 않고 contents 문서를 참조한다

## 문서 참조 지도
작업 성격을 파악하고 해당하는 문서를 읽은 뒤 작업한다.
두 가지 이상 해당하면 모두 읽는다.

| 작업 성격 | 읽어야 할 문서 |
|-----------|---------------|
| 서비스 방향, 기능 범위, Non-goal 판단 | @docs/ux/project-summary.md |
| 화면 구조, 플로우, IA, 인터랙션 | @docs/ux/ux-flow.md |
| 컴포넌트, 스타일, 토큰, 비주얼 | @docs/ux/design-system-optimization.md |
| 데이터 모델, API, 상태 관리, 라이브러리 | @docs/ux/technical.md |
| 텍스트, 메시지, 이미지, 미디어 | @docs/ux/contents.md |
```

### CLAUDE.md — 프로젝트 헌법

ux-architecture.md가 UX 판단을 담당하므로 CLAUDE.md는 더 짧아진다.
프로젝트 정체성과 기술 스택만 남긴다.

```markdown
# [프로젝트명]

## 핵심 가치
[한 문장]

## Non-goal
- [안 하는 것 1]
- [안 하는 것 2]

## 기술 스택
[프레임워크, 주요 라이브러리 한 줄 요약]

## 작업 전 확인
모든 작업 시작 전 .claude/rules/ux-architecture.md의 문서 참조 지도를 확인한다.
```

### 실제 동작 방식

세션이 시작되면 Claude는 자동으로 `ux-architecture.md`를 로딩한다.
작업 요청이 들어오면 지도를 보고 관련 `docs/ux/` 파일을 읽은 뒤 작업한다.

```
사용자: "마이페이지 화면 구현해줘"

Claude 내부 동작:
1. ux-architecture.md 지도 확인
2. "화면 구조" → @docs/ux/ux-flow.md 읽기
3. "컴포넌트/스타일" → @docs/ux/design-system-optimization.md 읽기
4. 두 문서 기준으로 구현
```

```
사용자: "에러 메시지 컴포넌트 만들어줘"

Claude 내부 동작:
1. ux-architecture.md 지도 확인
2. "텍스트/메시지" → @docs/ux/contents.md 읽기
3. "컴포넌트/스타일" → @docs/ux/design-system-optimization.md 읽기
4. 두 문서 기준으로 구현
```

### 이전 구조와 비교

```
이전                                    이후
──────────────────────────────────────────────────────
.claude/rules/에 5개 파일 자동 로딩   →  .claude/rules/에 1개만 자동 로딩
매 세션 5개 문서 전부 컨텍스트 점유   →  지도 1개만 상시 점유
작업과 무관한 문서도 항상 로딩        →  작업 성격에 따라 필요한 것만 읽기
문서 = 규칙 파일 (AI 전용)            →  문서 = 원본 (사람 + AI 공용)
```

### docs/ux/ 관리 원칙

- `docs/ux/` 파일은 사람이 읽고 수정하는 원본이다. 형식보다 내용이 중요하다
- `ux-architecture.md`의 참조 경로가 바뀌면 `docs/ux/` 파일명도 맞춰 업데이트한다
- 큰 결정이 바뀔 때(핵심 가치 변경, 컴포넌트 방향 전환)만 해당 문서를 업데이트한다
- Context compaction 전에 변경 사항을 `docs/ux/`에 먼저 반영해둔다
