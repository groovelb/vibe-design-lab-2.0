# Landing Funnel Redesign

> 피드백 기반 퍼널 스토리텔링 재설계안. 2026-03-12

---

## 1. 현재 퍼널 진단

| 섹션 | 현재 메시지 | 문제 |
|------|-----------|------|
| **HERO** | "바이브 코딩, 감으로 하지 않습니다" + 3 USP | 솔루션을 먼저 말함. 공감이 아니라 주장. 3 points가 Advantage 나열이지 Benefit이 아님 |
| **PROBLEM** | "시장이 요구하는 역할은 바뀌었지만..." | 추상적. 핵심 메시지("빌더가 되어야 합니다")가 묻혀있음 |
| **DIFFERENCE** | "도구 이전에, 언어 체계" | WHY 언어가 중요한지의 Benefit 부재. Advantage만 나열 |
| **→ METHOD** | (바로 연결) | Benefit 증거 없이 학습법으로 점프. "고치는 시간↓ 창조하는 시간↑"가 빠져있음 |

**핵심 문제:** PAS(Problem → Agitation → Solution) 순서가 깨져있다. Hero에서 이미 Solution을 말하고, Problem에서 뒤늦게 공감을 시도.

---

## 2. 재설계 퍼널 구조

```
HERO — 공감 + Pain (HOW 프레임)
│
PROBLEM — 위기감의 구체화
│
DIFFERENCE — 언어가 왜 중요한가
│
★ BENEFIT BRIDGE — 차이의 결과 증명 (신규)
│
METHOD — 어떻게 배우는가
│
COURSE → REVIEW → DICTIONARY → COMMUNITY → CTA
```

---

## 3. 섹션별 재설계

### 3.1 HERO — 공감 + Pain

**현재:** "바이브 코딩, 감으로 하지 않습니다" → VDL의 주장을 선언
**변경:** 유저의 현재 상황에 공감하는 질문형 Hook + Pain 직접 제시

| 요소 | 현재 | 재설계 방향 |
|------|------|-----------|
| Headline | "바이브 코딩, 감으로 하지 않습니다" | 공감형 질문. "바이브 코딩, 내맘대로 되고 있나요?" |
| SubCopy | "바이브 코딩에서 작동하는 디자인 언어 체계를 배웁니다" | Solution 제거. Pain 공감으로 |
| 3 Points | Advantage 나열 (의미전달/키워드/구현) | **Pain을 있는 그대로** |

#### 3 Pain Points 재설계

| 현재 (Advantage) | 재설계 (Pain) |
|-----------------|--------------|
| 정확한 의미 전달 — UX 설계로 구조화된 의도를 전달합니다 | (예) 프롬프트가 길어질수록 산으로 갑니다 |
| 정밀한 디자인 키워드 — 디자인 택소노미로 AI 클리셰를 벗어납니다 | (예) AI가 알아듣는 단어를 써야합니다 |
| 디자인 중심 구현 — 디자인 자체로 구현의 설계도를 완성합니다 | (예) 고치는 시간이 만드는 시간보다 깁니다 |

#### 페르소나 네이밍

현재 페르소나 라벨("캔버스 디자이너", "샌드위치 PM", "파이프라인 개발자")이 Hero에는 등장하지 않고 Problem에서만 사용됨. Hero에서 페르소나를 **풀어서 설명**하는 방식 검토 필요.

- 라벨만으로는 의미 전달이 안 됨 → "캔버스 디자이너"가 무슨 뜻인지 직관적이지 않음
- Hero에서 페르소나 상황을 한 줄로 묘사하거나, Pain 3 Points 자체가 페르소나별 pain이 되는 구조

---

### 3.2 PROBLEM — 위기감의 구체화

**현재:** "시장이 요구하는 역할은 바뀌었지만, 실무자들은 점점 따라가기 어렵습니다."
**변경:** 핵심 메시지를 전면에 배치

#### 타이틀 후보

현재 숨겨진 메시지를 꺼내야 함:

> "생각하는 사람이 직접 만들 수 있어야 합니다"

⚠️ "AI로 대체되지 않기 위해"는 `TONE_BLACKLIST.fearMarketing`("AI에 대체됩니다", "살아남으려면") 경계선.
VDL 톤에 맞는 방향: 공포가 아닌 **방향 제시**.

| 톤 | 표현 | 판정 |
|---|------|-----|
| 공포 마케팅 | "대체되지 않으려면 빌더가 되어야 합니다" | ✕ blacklist 위반 |
| 방향 제시 | "빌더가 되어야 합니다" | △ 단정적이나 공포 뉘앙스 잔존 |
| 미션 연결 | "생각하는 사람이 직접 만들 수 있어야 합니다" | ○ VDL 미션 정렬 |

#### 페르소나 카드

현재 구조(ScrollRevealText + StatementCard)는 유지. 카드 내 워딩만 조정.

---

### 3.3 DIFFERENCE — 언어의 Benefit

**현재:** 3 Value Propositions를 Advantage로 나열
**변경:** 언어 체계가 주는 **Benefit**을 구체적으로 설명

| 요소 | 현재 | 재설계 방향 |
|------|------|-----------|
| Headline | "도구 이전에, 언어 체계" | 아래 후보 참조 |
| SubCopy | "도구는 바뀌어도 설계의 기준은 남습니다" | 언어 체계가 주는 구체적 Benefit |
| 3 Cards | Advantage (System Over Drawing / The Vibe Standard / Design As Build) | Benefit 중심 리프레임 |

#### Headline 후보

| 후보 | 출처 | 비고 |
|------|------|------|
| ~~"바이브 코딩은 말이 중요합니다"~~ | 피드백 원문 | ✕ 구체성 부족. 어떤 비즈니스든 "말이 중요하다"고 할 수 있음 |
| "구현은 언어를 따른다" | Philosophy ① | VDL 핵심 신념. 단정적, 리듬감 있음 |
| "정밀하게 말하면, 정밀하게 나온다" | The Vibe Standard 숏버전 | Benefit이 문장 안에 내포됨 |
| "결과를 바꾸는 건 도구가 아니라 언어입니다" | Myth 1 반전 파생 | 설명적, 대비 구조 |

"구현은 언어를 따른다" + sub "정밀하게 말하면, 정밀하게 나온다" 조합이 유력.
Headline이 원리(왜), SubCopy가 Benefit(그래서 뭐).

#### 3 VP의 Benefit 리프레임

| VP | 현재 (Advantage) | Benefit 방향 |
|---|-----------------|-------------|
| System Over Drawing | 결과물보다 기준을 먼저 설계합니다 | 기준이 있으면 AI가 흔들리지 않는다 |
| The Vibe Standard | AI가 알아듣는 표준 디자인 언어 체계 | 정밀하게 말하면 정밀하게 나온다 |
| Design As Build | 구현의 설계도가 되는 디자인 접근방식 | 설계의 상세도가 제품의 완성도를 결정한다 |

---

### 3.4 ★ BENEFIT BRIDGE (신규 섹션)

**위치:** DIFFERENCE → METHOD 사이
**목적:** Difference에서 말한 "언어 체계"의 실제 결과를 증명

#### 핵심 메시지

> "고치는 시간을 줄이고, 원하는 걸 창조하는 시간이 많아진다"

#### 구성 요소

| 요소 | 설명 |
|------|------|
| **프로세스 시각화** | 언어 없이 바이브코딩 vs 체계적 언어로 바이브코딩. 분할 화면 인터랙션 |
| **바이브 코딩 쇼케이스** | 화려한 실제 결과물 예시. "나도 저런걸 만들 수 있구나" — 시각적 퍼포먼스 |
| **구체적 예시** | 언어는 추상적 → 분할화면에서 체계적인 프롬프트와 결과를 대비 |

#### 쇼케이스 위치 판단

| 위치 | 장점 | 단점 |
|------|------|------|
| Hero 하단 | 첫 인상에서 "와" 효과 | "왜"를 설명하기 전에 "결과"를 보여줌. 인과관계 약함 |
| **Benefit Bridge** | Difference("언어가 중요하다") 직후 → 인과관계 성립 | Hero에서의 즉시 임팩트는 포기 |

**결론:** Benefit Bridge 쪽이 적합. "언어가 중요하다"(Difference) → "이런 결과가 나온다"(Bridge) 순서가 설득의 인과관계를 만든다.

## 5. 톤 가이드라인 체크

### 적용 기준 (조정됨)

- **정밀**(빈 문장 금지) — **엄격 유지**. "말이 중요합니다"처럼 아무 비즈니스에나 쓸 수 있는 문장은 ✕
- **주도**(선언형) — **Hero 공감 구간에서 완화**. 질문형, 구어체("내맘대로") 허용
- **블랙리스트** — 유지. 공포 마케팅, 허위 약속 금지

### 제약 목록

| 규칙 | 적용 |
|------|------|
| `TONE_BLACKLIST.fearMarketing` | "대체됩니다", "살아남으려면" 사용 금지 |
| `TONE_BLACKLIST.falsePromise` | "누구나 할 수 있다" 사용 금지 |
| `VOICE_RULES.주도` | 기본 단정형. **Hero 공감 구간에서 질문형/구어체 허용** |
| `VOICE_RULES.정밀` | 추상적 형용사, 빈 선언 금지. 구체적 키워드·현상 묘사 필수 |
| `VOICE_RULES.절제` | 과장하지 않는다. 결과와 근거로 말한다 |
| `POSITIONING[2]` | "누구나 할 수 있다" ✕ → "잘하려면 디자인 언어가 필요하다" ○ |

### 제안 카피 톤 점검

| 카피 | 판정 | 비고 |
|------|:----:|------|
| "바이브 코딩, 내맘대로 되고 있나요?" | ○ | 구어체지만 구체적 pain을 짚음. Hero 공감 구간 허용 |
| "프롬프트가 길어질수록 산으로 갑니다" | ○ | 구어체 허용 범위. 현상이 구체적 |
| "AI가 알아듣는 단어를 써야합니다" | ○ | 직접적, 행동 지향 |
| "고치는 시간이 만드는 시간보다 깁니다" | ○ | 측정 가능한 대비 |
| ~~"바이브 코딩은 말이 중요합니다"~~ | ✕ | 톤이 아니라 **구체성**의 문제. 빈 선언 |
| "구현은 언어를 따른다" | ○ | Philosophy 원문. 단정적, 구체적 |

## 6. 작업 순서 (제안)

1. `contents.js` 메시지 재설계 — Hero pain points, Problem headline, Difference benefit
2. `LandingBenefitBridge` 신규 섹션 설계 + 컴포넌트 구조 정의
3. Hero 구조 수정 — 공감형 hook + pain 3 points
4. Difference 리프레임 — Advantage → Benefit
5. 쇼케이스 인터랙션 설계 (별도)
