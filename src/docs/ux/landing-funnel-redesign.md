# Landing Funnel Redesign

> 피드백 기반 퍼널 스토리텔링 재설계안. 2026-03-12 (v2 — 확정)

---

## 1. 현재 퍼널 진단

| 섹션 | 현재 메시지 | 문제 |
|------|-----------|------|
| **HERO** | "바이브 코딩, 감으로 하지 않습니다" + 3 USP | 솔루션을 먼저 말함. 공감이 아니라 주장. 3 points가 Advantage 나열이지 Benefit이 아님 |
| **PROBLEM** | "시장이 요구하는 역할은 바뀌었지만..." | 추상적. 핵심 메시지("빌더가 되어야 합니다")가 묻혀있음 |
| **DIFFERENCE** | "도구 이전에, 언어 체계" | WHY 언어가 중요한지의 Benefit 부재. Advantage만 나열. PROBLEM → METHOD 사이 인과 약함 |

**핵심 문제:** PAS(Problem → Agitation → Solution) 순서가 깨져있다. Hero에서 이미 Solution을 말하고, Problem에서 뒤늦게 공감을 시도. DIFFERENCE가 Solution이 아닌 Advantage 나열에 그침.

---

## 2. 재설계 퍼널 구조

```
HERO — 공감 + Pain
│
PROBLEM — 방향 제시 ("빌더가 앞서갑니다")
│
WHY VIBE DESIGN — 편견 뒤집기 (Agitation)
│
SOLUTION — 빌더가 되는 방법 = 언어 체계 (기존 DIFFERENCE → 리네이밍)
  └ 쇼케이스: 바이브 코딩 결과물 (기존 BENEFIT BRIDGE 흡수)
│
METHOD — 어떻게 배우는가
│
COURSE → REVIEW → DICTIONARY → COMMUNITY → CTA
```

**변경점:** DIFFERENCE + BENEFIT BRIDGE 2개 섹션 → SOLUTION 1개로 통합. PROBLEM의 "빌더가 되어야 한다"에 대한 직접 응답("빌더의 도구 = 언어 체계")으로 연결. 쇼케이스는 SOLUTION 하단에 증거로 배치.

## 3. 섹션별 확정 카피

### 3.1 HERO — 공감 + Pain

| 요소 | 현재 | 확정 |
|------|------|------|
| Headline | "바이브 코딩, 감으로 하지 않습니다" | **"바이브 코딩, 내맘대로 되고 있나요?"** |
| SubCopy | "바이브 코딩에서 작동하는 디자인 언어 체계를 배웁니다" | **제거** (Pain Points가 역할 대체) |

#### 3 Pain Points

| # | 현재 (Advantage) | 확정 (Pain) | 작동 원리 |
|---|-----------------|------------|----------|
| 1 | 정확한 의미 전달 — UX 설계로 구조화된 의도를 전달합니다 | **프롬프트가 길어질수록 산으로 갑니다** | 노력↑ 결과↓ 괴리 |
| 2 | 정밀한 디자인 키워드 — 디자인 택소노미로 AI 클리셰를 벗어납니다 | **원하는 걸 AI에게 설명이 안 됩니다** | 의도-표현 갭 |
| 3 | 디자인 중심 구현 — 디자인 자체로 구현의 설계도를 완성합니다 | **고치는 시간이 만드는 시간보다 깁니다** | 시간 낭비 |

> PP 순서가 경험 흐름의 escalation을 따름: 말이 길어짐(1) → 전달 자체가 안 됨(2) → 결국 고치기만 함(3).

---

### 3.2 PROBLEM — 방향 제시

| 요소 | 현재 | 확정 |
|------|------|------|
| Headline | "시장이 요구하는 역할은 바뀌었지만..." | **"직장인이건 사업자건, 빌더가 앞서갑니다"** |

"빌더가 앞서갑니다" — 공포("대체됩니다") 없이, 방향과 긍정적 동기를 제시. "앞서갑니다"는 선두를 의미하되 공포를 유발하지 않음.

#### 페르소나 카드

구조(ScrollRevealText + StatementCard) 유지. subtitle 워싱:

| 페르소나 | 현재 subtitle | 확정 subtitle |
|---------|-------------|-------------|
| 캔버스 디자이너 | 피그마 벗어나지 못하는 존재감 | **캔버스 밖에서 존재감이 없습니다** |
| 샌드위치 PM | 보여줄 게 없어서 말로만 설득하는 한계 | **보여줄 게 없으면 말로는 한계입니다** |
| 파이프라인 개발자 | 작동은 하지만 왜 어색한지 모르는 답답함 | (유지) |

> subtitle을 상황 묘사 → 결과 진술로 변경. VDL "주도" 보이스(단정형)에 정렬.

#### 학습 한계 (유지)

```
VOD 한계: 혼자 듣는 VOD는 혼자 막히고 혼자 포기한다
도구 종속: 도구 사용법을 배워도 도구가 바뀌면 처음부터다
학습 단절: 코스가 끝나면 끝이다. 쌓이는 게 없다
```

### 3.3 WHY VIBE DESIGN — 편견 뒤집기

**Statement (유지):**
> "그동안 디자인은 과소평가 당해왔습니다. 연출이 아닌 설계로서의 디자인은, 바이브 코딩에서 가장 정확한 표현 수단입니다."

PROBLEM("빌더가 앞서갑니다") → WHY("디자인이 과소평가 당해왔다") → SOLUTION("디자인 언어가 빌더의 도구")의 징검다리.

**3 Myths (유지):**

| # | 편견 (conventional) | 뒤집기 (reversal) |
|---|-------------------|-----------------|
| 1 | 더 좋은 AI 모델이 나오면 결과물이 좋아진다 | 결과를 바꾸는 건 모델의 성능이 아니라 내 의도를 날카롭게 깎는 훈련입니다 |
| 2 | 일단 만들고, 디자인은 나중에 입히면 된다 | 디자인은 설계와 구현을 동시에 담당하는 언어입니다 |
| 3 | 새 스킬을 빨리 배워야 살아남는다 | 도구가 바뀌어도 작동하는 건 경험이 축적된 언어 체계입니다 |

> Myth #3 "살아남는다"는 VDL 목소리가 아닌 **뒤집히는 편견**(세간의 말)이므로 blacklist 적용 외.

---

### 3.4 SOLUTION (기존 DIFFERENCE + BENEFIT BRIDGE 통합)

**연결 논리:**
```
PROBLEM: "빌더가 앞서갑니다"
   ↓ 빌더가 되려면?
SOLUTION: "구현은 언어를 따른다"
   = 언어 체계가 빌더의 도구
```

| 요소 | 현재 | 확정 |
|------|------|------|
| Headline | "도구 이전에, 언어 체계" | **"구현은 언어를 따른다"** |
| SubCopy | "도구는 바뀌어도 설계의 기준은 남습니다" | **"정밀하게 말하면, 정밀하게 나온다"** |

> Headline = 원리(왜), SubCopy = Benefit(그래서 뭐). Philosophy ① 원문 활용.

#### 3 VP Cards — Benefit 리프레임

| # | Tab | 현재 Headline (Advantage) | 확정 Headline (Benefit) | 확정 설명 |
|---|-----|-------------------------|----------------------|----------|
| 1 | Design Context | 디자인 산출물이 AI의 컨텍스트가 된다 | **기준이 있으면 AI가 흔들리지 않는다** | 기획서·UX 플로우·레퍼런스 보드가 AI의 설계 맥락이 됩니다. 의도와 구현 사이의 거리가 줄어듭니다. |
| 2 | Design Taxonomy | 이름을 붙이면 의도가 전달된다 | **이름을 붙이면, 결과가 일관됩니다** | 히어로 섹션, 카드 그리드, Bento 레이아웃 — 명명된 패턴은 AI에게도 동료에게도 같은 결과를 만듭니다. |
| 3 | Meta-language | 설계의 상세도가 제품의 완성도를 결정한다 | **설계의 상세도가 제품의 완성도를 결정한다** (유지) | UX 시나리오의 깊이가 데이터 모델과 API 구조를 결정합니다. 디자인은 화면의 일부가 아니라 제품 전체를 설계하는 메타언어입니다. |

> VP#2: "의도가 전달된다"(Advantage) → "결과가 일관됩니다"(Benefit). SubCopy와의 중복도 해소.

#### 쇼케이스 (SOLUTION 하단)

VP 카드 아래에 증거로 배치. 별도 섹션이 아닌 SOLUTION 안에 포함.

| 요소 | 설명 |
|------|------|
| **바이브 코딩 결과물 갤러리** | "이 언어 체계로 만들었습니다" — 실제 결과물 시각적 임팩트 |
| **분할 화면 비교** | 언어 없이 바이브코딩 vs 체계적 언어로 바이브코딩 |

> "이런 원리입니다"(VP 카드) → "이런 결과입니다"(쇼케이스) 인과가 한 섹션 안에서 완결.

### 3.5 METHOD 이후 — 변경 없음

| 섹션 | Headline | 상태 |
|------|----------|------|
| METHOD | 리드와 참여자가 함께 만들어갑니다 | 유지 |
| COURSE | 코스 | 유지 |
| REVIEW | 많은 실무자들이 검증한 커리큘럼입니다 | 유지 |
| DICTIONARY | 도구가 바뀌어도 남는 체계 | 유지 |
| COMMUNITY | 지금 커뮤니티에서 일어나는 일 | 유지 |
| CTA | 시작합니다 | 유지 |

## 5. 톤 가이드라인 체크

### 적용 기준 (조정됨)

- **정밀**(빈 문장 금지) — **엄격 유지**. "말이 중요합니다"처럼 아무 비즈니스에나 쓸 수 있는 문장은 ✕
- **주도**(선언형) — **Hero 공감 구간에서 완화**. 질문형, 구어체("내맘대로") 허용
- **블랙리스트** — 유지. 공포 마케팅, 허위 약속 금지

### 제약 목록

| 규칙 | 적용 |
|------|------|
| `TONE_BLACKLIST.fearMarketing` | "대체됩니다", "살아남으려면" 사용 금지 |
| `TONE_BLACKLIST.falsePromise` | "누구나 할 수 있다" 사용 금지 |
| `VOICE_RULES.주도` | 기본 단정형. **Hero 공감 구간에서 질문형/구어체 허용** |
| `VOICE_RULES.정밀` | 추상적 형용사, 빈 선언 금지. 구체적 키워드·현상 묘사 필수 |
| `VOICE_RULES.절제` | 과장하지 않는다. 결과와 근거로 말한다 |
| `POSITIONING[2]` | "누구나 할 수 있다" ✕ → "잘하려면 디자인 언어가 필요하다" ○ |

### 확정 카피 톤 점검

| 카피 | 판정 | 비고 |
|------|:----:|------|
| "바이브 코딩, 내맘대로 되고 있나요?" | ○ | 구어체지만 구체적 pain을 짚음. Hero 공감 구간 허용 |
| "프롬프트가 길어질수록 산으로 갑니다" | ○ | 구어체 허용 범위. 현상이 구체적 |
| "원하는 걸 AI에게 설명이 안 됩니다" | ○ | 직접적 pain. 의도-표현 갭 묘사 |
| "고치는 시간이 만드는 시간보다 깁니다" | ○ | 측정 가능한 대비 |
| "직장인이건 사업자건, 빌더가 앞서갑니다" | ○ | 긍정적 방향 제시. 공포 없이 동기 부여 |
| "캔버스 밖에서 존재감이 없습니다" | ○ | 단정형 결과 진술. 주도 보이스 정렬 |
| "보여줄 게 없으면 말로는 한계입니다" | ○ | 직접적, 측정 가능 |
| "구현은 언어를 따른다" | ○ | Philosophy 원문. 단정적, 구체적 |
| "정밀하게 말하면, 정밀하게 나온다" | ○ | Benefit 내포. 인과 구조 |
| "기준이 있으면 AI가 흔들리지 않는다" | ○ | 조건-결과 구조. 구체적 |
| "이름을 붙이면, 결과가 일관됩니다" | ○ | 행동-결과 대비 |
| ~~"바이브 코딩은 말이 중요합니다"~~ | ✕ | 구체성 부족. 빈 선언 |

## 6. 내러티브 아크

```
HERO        "내맘대로 되고 있나요?"              ← 당신의 현실
  │          3 Pain Points
  │
PROBLEM     "빌더가 앞서갑니다"                 ← 왜 바꿔야 하는가
  │          페르소나별 구체적 pain
  │
WHY VDL     "디자인은 과소평가 당해왔습니다"      ← 잘못된 믿음 깨기
  │          3 Myths reversed
  │
SOLUTION    "구현은 언어를 따른다"               ← 빌더가 되는 방법
  │          3 Benefits + 쇼케이스(증거)
  │
METHOD      "함께 만들어갑니다"                  ← 우리가 가르치는 방식
  │
COURSE → REVIEW → CTA                         ← 전환
```

각 섹션 → 다음 섹션의 자연스러운 질문에 답:
- "안 되고 있다" → "왜?" → "잘못 알고 있었다" → "이렇게 하면 된다" → "이렇게 배운다" → "시작하자"

## 7. 작업 순서 (제안)

1. `contents.js` 메시지 업데이트 — Hero PP, Problem headline, SOLUTION 카피
2. `howDifferent` → SOLUTION 리네이밍 + VP Benefit 리프레임
3. Hero 구조 수정 — subCopy 제거 + Pain Points 교체
4. Problem headline + 페르소나 subtitle 워싱
5. 쇼케이스 영역 설계 (별도)
