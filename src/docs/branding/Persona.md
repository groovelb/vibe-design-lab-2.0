## 바이브 코딩 소구 프레임워크: 세 페르소나에게 바이브 코딩이 의미하는 것

### 전제: 바이브 코딩은 "코딩을 쉽게 하는 것"이 아니다

바이브 코딩의 본질은 구현(Implementation)에서 지휘(Orchestration)로의 전환이다. 카파시의 정의처럼 "코드가 존재한다는 사실조차 잊어버리는 것"— 이건 기술적 편의가 아니라 **본업의 확장**을 뜻한다.

디자이너는 디자이너로 남으면서 구현까지 맡고, PM은 PM으로 남으면서 검증까지 하고, 개발자는 개발자로 남으면서 제품 의사결정까지 한다. 직업을 바꾸는 게 아니라 **자기 본업의 반경을 AI로 넓히는 것** — 이것이 바이브 코딩이 세 페르소나에게 소구하는 근본 이유다.

### 소구 프레임워크 템플릿

세 페르소나의 소구 구조는 하나의 공통 템플릿을 따른다:

```
[본업] — 이미 가지고 있는 것
    ↓ 바이브 코딩 + VDL
[본업 + 확장] — AI로 빠진 절반을 채움
    ↓
[채용시장 경쟁력] — 중간 단계. 기존 고용 구조 안에서 입지 강화
    ↓
[채용 구조 독립] — 최종 단계. 고용에 의존하지 않는 독립적 주체

※ 이 두 단계는 별개의 솔루션이 아니라 같은 스펙트럼 위에 있다.
   본업을 확장하면 채용시장에서 살아남고, 충분히 확장하면 채용 구조 자체에서 독립한다.
```

### 페르소나별 소구 구조

**캔버스 디자이너 — 디자인의 동작과 구현까지 맡는 디자이너**

| 축 | 내용 |
|---|---|
| 본업 (이미 있는 것) | 디자인 판단력 — 무엇이 좋은 UI/UX인지 안다 |
| 빠진 것 | 동작과 구현 — 판단한 것을 코드로 실현하는 능력 |
| 바이브 코딩이 주는 것 | 디자인 의도를 끝까지 통제하는 힘. Figma 안의 정적인 결정을 동작하는 결과물로 완결 |
| 채용시장 경쟁력 | JD 범위 확장. "디자인 엔지니어" 포지션 지원 가능. 연봉 협상 레버리지 |
| 채용 구조 독립 | 디자인+구현+배포를 혼자 완결 → 프리랜서 건당 단가 상승, 독립 프로덕트 가능 |

**샌드위치 PM — 결과물로 의사결정하는 PM**

| 축 | 내용 |
|---|---|
| 본업 (이미 있는 것) | 제품 전략과 판단력 — 무엇을 만들어야 하는지 안다 |
| 빠진 것 | 검증과 시연 — 판단한 것을 직접 만들어서 보여주는 능력 |
| 바이브 코딩이 주는 것 | 말이 아니라 결과물로 설득하는 힘. "이 기능 3주 걸린다"를 직접 시연해서 뒤집는 주도권 |
| 채용시장 경쟁력 | "기획만 하는 PM"에서 파이프라인 전체를 읽고 판단하는 PM으로. 조직 내 입지, 승진 레버리지 |
| 채용 구조 독립 | 리서치+스펙+프로토타입을 혼자 완결 → 독립적 제품 검증, 사내 벤처 리드 가능 |

**파이프라인 개발자 — 제품 전체의 의사결정이 가능한 개발자**

| 축 | 내용 |
|---|---|
| 본업 (이미 있는 것) | 구현 능력 — 코드를 쓸 줄 안다 |
| 빠진 것 | 제품 의사결정 언어 — 왜 이 레이아웃인지, 왜 이 간격인지, 왜 이 플로우인지 결정하는 기준 |
| 바이브 코딩이 주는 것 | 티켓 바깥에서 제품을 설계하는 힘. 구현자에서 의사결정자로의 확장 |
| 채용시장 경쟁력 | "풀스택 + 제품 감각" 포지션. 티켓 실행자가 아니라 제품 전체를 다루는 개발자로 평가 |
| 채용 구조 독립 | 제품 기획+디자인 결정+구현을 혼자 완결 → 솔로프리너, 인디 해커, 프로덕트 단위 수익화 |

### 공통 구조: 본업 확장 → 채용 경쟁력 → 채용 독립

```
Before Vibe Coding:
캔버스 디자이너  →  디자인은 하지만 구현 의존     →  캔버스 안에 갇힘
샌드위치 PM     →  전략은 있지만 검증 의존       →  위아래에 끼임
파이프라인 개발자 →  구현은 하지만 의사결정 언어 부재 →  티켓에 갇힘

After Vibe Coding + VDL:
캔버스 디자이너  →  디자인 + 동작·구현 확장      →  의도를 끝까지 통제
샌드위치 PM     →  전략 + 검증·시연 확장        →  결과물로 의사결정
파이프라인 개발자 →  구현 + 제품 의사결정 확장     →  제품 전체를 설계
```

세 사람의 구조는 같다: **본업은 유지하되, AI로 빠진 절반을 채워서 본업의 반경을 확장한다.** 그 확장이 채용시장 안에서는 경쟁력이 되고, 충분히 확장되면 채용 구조 자체에서 독립하는 힘이 된다. 두 결과는 같은 과정의 다른 단계일 뿐이다.

### VDL은 여기서 무엇을 하는가

바이브 코딩 도구(v0, Cursor, Lovable)는 이미 존재한다. 하지만 도구만으로는 독립이 안 된다. 시장 분석에서 확인된 사실— 주요 플랫폼 중 명시적 패턴 분류 체계를 노출하는 곳이 없고, 모두 암묵적 LLM 패턴 인식에 의존하며, 디자인 어휘를 이미 아는 사용자에게만 보상한다. 도구는 있는데 **도구를 제대로 지휘하는 언어가 없는 상태**.

VDL은 도구가 아니라 그 도구 위에서 **독립적 주체가 되기 위한 리터러시**를 제공한다.

| VDL 자산 | 기능적 역할 | 커리어적 역할 |
|---------|-----------|-------------|
| **Vibe Dictionary** | AI가 알아듣는 디자인 언어 체계 | 결과물 품질 = 단가/연봉의 근거 |
| **교육 (코호트)** | 의도를 체계적으로 정의하고 전달하는 역량 | 포트폴리오에 배포 URL이 추가되는 8주 |
| **커뮤니티** | 독립적 주체들의 상호 검증 네트워크 | 채용 기회, 협업 기회, 레퍼런스 |

### 학습 환경의 구조적 문제 — 기존 교육이 실패하는 이유

커리어 페인포인트(아래 각 페르소나 참조)가 학습 동기를 만들지만, 학습 환경이 구조적으로 실패하면 동기가 소멸한다. VDL은 "왜 배워야 하는가"(커리어 문제)와 "어떻게 배워야 하는가"(학습 환경 문제)를 동시에 해결한다.

| 문제 | 원인 | VDL의 접근 |
|------|------|-----------|
| **정적 VOD의 한계** | 혼자 듣고, 혼자 막히고, 혼자 포기한다. 질문은 맥락 없이 게시판에 묻히고, 동료의 시행착오에서 배울 수 없다. 완주율이 낮고, 완주해도 실전에서 쓰지 못한다 | 코스가 커뮤니티처럼 운영된다. 챕터별 질문과 챌린지가 실시간 공개되고, 멤버 간 결과물이 공유되며, 동료 압력이 학습 동력이 된다. 플랫폼 + Discord 연동으로 비동기 아카이브와 실시간 소통을 모두 지원 |
| **도구 종속적 학습** | v0, Cursor 사용법을 배워도 6개월 뒤 도구가 바뀌면 처음부터. 시장의 대부분의 교육이 도구 사용법에 집중하고, 도구 위에서 작동하는 언어를 다루지 않는다 | 한 번 배우면 계속 가는 디자인 언어 체계를 가르친다. 코스의 본체는 메타 지식(디자인 원리, 시스템 사고)이고, 도구 예제는 연 2회 업데이트되는 껍데기. Dictionary가 이 체계의 산출물 |
| **학습 이후 단절** | 코스가 끝나면 끝. 졸업생의 성장이 다음 학습자에게 전달되지 않는다. 교육 → 결과 → 마케팅 → 유입의 순환이 없다 | 멤버가 운영·마케팅에 연계되는 순환 생태계. 기존 멤버 요청에 따라 연 2회 코스 업데이트. 졸업생 증언이 다음 Attract 자산. 졸업생이 커뮤니티에 남아 피어 리뷰어가 되고, 그 순환 자체가 해자 |

**학습 환경이 페르소나별로 다르게 작동하는 이유:**
- **캔버스 디자이너**: 코딩이 완전히 새로운 영역이라 혼자 막히면 바로 포기. 동료 디자이너의 시행착오를 보면서 "나만 어려운 게 아니구나"를 확인해야 계속 간다
- **샌드위치 PM**: 현업 병행 중이라 시간 압박이 크고, 자기 업무에 바로 적용할 수 있는 맥락이 필요. 다른 PM의 실무 적용 사례가 동기부여
- **파이프라인 개발자**: 코딩은 되니까 기술적 탈락은 적지만, "디자인 감각"이라는 막연한 영역을 체계로 분해하는 과정에서 동료의 관점이 결정적. 다른 개발자가 "이게 체계였어?"라고 깨닫는 순간을 함께 보면서 확신이 강해진다

## 01. 캔버스 디자이너

### 프로파일

| 항목 | 상세 |
|------|------|
| 직군 | UX 디자이너, 프로덕트 디자이너, UI 디자이너 |
| 경력 | 3~8년차. 도구는 능숙하지만 영향력의 천장을 느끼는 시점 |
| 핵심 도구 | Figma, Sketch, Adobe XD |
| 채널 | Threads, X, 유튜브, Instagram |
| 한 줄 독백 | "내가 만든 디자인인데, 완성은 항상 남의 몫이다" |

### As-is → To-be

```
As-is: 캔버스 디자이너
┌──────────────────────────────────┐
│  Figma 안에서만 존재하는 영향력     │
│  넘기면 의도가 왜곡됨              │
│  산출물 = 목업, 프로토타입          │
│  "이건 개발팀에 넘겨야 해요"        │
└──────────────────────────────────┘
              ▼  VDL
To-be: 프로덕트 엔지니어
┌──────────────────────────────────┐
│  코드 + 배포까지의 전체 결정권      │
│  내 의도가 그대로 구현됨            │
│  산출물 = 배포된 앱                │
│  "제가 배포했습니다"               │
└──────────────────────────────────┘
```

### 페인포인트 상세

| 순위 | 페인포인트 | 빈도 | 심각도 |
|------|-----------|------|--------|
| 1 | 채용 시장이 디자인 엔지니어를 요구하는데 나는 Figma만 할 줄 안다 | 매일 (JD 볼 때마다) | ★★★★★ |
| 2 | AI 도구에 "예쁘게 만들어줘"밖에 못 써서 매번 결과가 다르다 | 매번 (v0, Cursor 사용 시) | ★★★★☆ |
| 3 | 개발팀에 넘기면 간격, 색상, 트랜지션이 의도와 달라진다 | 매 스프린트 | ★★★★☆ |
| 4 | 코딩 VOD를 혼자 듣다가 막히면 질문할 곳도 동료도 없어서 바로 포기한다 | 학습 시도마다 | ★★★★☆ |
| 5 | 도구 튜토리얼(v0, Cursor)을 따라해도 도구가 바뀌면 처음부터 — 배운 게 쌓이지 않는다 | 도구 업데이트마다 | ★★★★☆ |
| 숨겨진 | 자기 디자인 결정의 근거를 논리적으로 설명하지 못한다 | 인식 못 함 | ★★★★★ |

숨겨진 페인포인트가 가장 중요한 이유: 이게 세 층위 ③(디자인 본질 회귀)의 진입점. "이게 좋아 보여서"라고 말하던 사람이 "이건 Glassmorphism이고, 이 맥락에서 이 패턴을 쓰는 이유는..."이라고 말할 수 있게 되는 전환. 본인은 이게 페인이라는 걸 모르고 있다가, 모듈 2(토큰 매핑 실습)에서 처음 자각함.

### 상승 욕구

**욕망의 3단계 구조:**

```
도구적 욕망        "v0, Cursor를 쓸 줄 알고 싶다"
    ↓
결과적 욕망        "내 디자인을 직접 배포하고 싶다"
    ↓
정체성 욕망        "나는 프로덕트 엔지니어다"
```

- 욕망 강도: ★★★★★
- 채용 보고서 근거: Vercel JD — "디자인 리더십과 엔지니어링 탁월함의 결합", Clarify.ai — "프로덕션 코드 직접 배포"
- 구조적 뒷받침: 채용 시장이 핸드오프 제거 방향으로 이동 중. 일시적 트렌드가 아니라 커리어 생존과 연결

### 불안 요소

| 불안 유형 | 구체적 형태 | 강도 |
|----------|------------|------|
| 존재적 불안 | "v0가 UI를 만들어주는데 내가 왜 필요하지?" | ★★★★★ |
| 역량 불안 | "150~250만원 쓰고 코딩 배워도 진짜 구현까지 할 수 있을까?" | ★★★★☆ |
| 정체성 불안 | "코딩을 배우면 디자이너가 아니게 되는 건가?" | ★★★☆☆ |

VDL의 해소 전략:
- 존재적 불안 → 층위 ②로 구조적 해소: "AI가 실행을 가져갈수록, 의도를 정의하는 너의 역할이 강해진다"
- 역량 불안 → 퍼널로 해소: Dictionary(무료) → 프롬프트 클리닉(무료) → 48시간 빌드 챌린지(무료 체험) → 코호트
- 정체성 불안 → 층위 ③이 자연스럽게 해소: "코드를 배우니까 오히려 디자인을 더 정확히 말하게 됐다"

### 세 층위 반응

| 층위 | 이 사람에게 어떻게 작동하는가 | 활성 시점 |
|------|------------------------------|----------|
| ① 커리어 상승 | "Vercel이 뽑는 사람은 디자인 엔지니어다. 나는 Figma만 할 줄 안다." → 채용 JD가 욕망과 위기감을 동시에 자극 | Attract |
| ② 대체 불안 해소 | "v0가 UI를 만들어주는데 나는 뭘 해야 하지?" → "AI에게 정확히 말하는 능력이 내 역할이다" | Engage + 모듈 1 |
| ③ 디자인 본질 회귀 | "토큰을 정의하고 .cursorrules를 쓰면서, 왜 이 간격이 8px이어야 하는지 처음으로 설명할 수 있게 됐다" | 모듈 2~3 + 결과물 발표일 |

③ 도달 가능성: **높음** — 세 페르소나 중 가장 높음. 디자인 결정을 매일 하는 사람이기 때문에, 그 결정을 언어화하는 훈련이 직접적으로 와닿음.

### 상승 욕구

**욕망의 핵심: 수단의 확보**

캔버스 디자이너의 욕망이 "정체성 전환"이라면, 샌드위치 PM의 욕망은 더 실용적 — "월요일 회의에서 보여줄 수 있는 것". 코드를 배우고 싶은 게 아니라, **만들어서 보여주는 능력**이 필요한 것.

- 욕망 강도: ★★★★☆
- 채용 보고서 근거: PM → 프로덕트 오케스트레이터 전환. 산출물이 "목업, PRD"에서 "배포된 앱"으로 이동
- 구매 동기: 즉각적 ROI를 따짐. "이번 분기에 바로 쓸 수 있는가?"

- 존재적 불안 → "무엇을 만들지 정의하고, AI+팀을 지휘하는 것이 네 역할" — 오케스트레이터 재정의
- 역할 불안 → 48시간 빌드 챌린지로 "나도 할 수 있다" 선체험
- 시간 불안 → B2B 워크샵 경로 제공 (코호트 대안)

## 03. 파이프라인 개발자

### 프로파일

### As-is → To-be

```
As-is: 파이프라인 개발자
┌──────────────────────────────────┐
│  할당된 티켓 범위 안에서만 일함     │
│  디자인은 남이 결정, 나는 구현      │
│  제품 전체를 다루는 언어가 없음     │
│  UI는 "돌아가면 됨"               │
└──────────────────────────────────┘
              ▼  VDL
To-be: 제품 의사결정이 가능한 개발자
┌──────────────────────────────────┐
│  파이프라인 바깥에서 제품을 설계함   │
│  디자인 결정의 근거를 읽고 판단함   │
│  구현뿐 아니라 "왜 이렇게"를 결정함 │
│  UI는 "의도를 구조화한 것"         │
└──────────────────────────────────┘
```

### 페인포인트 상세

| 순위 | 페인포인트 | 빈도 | 심각도 |
|------|-----------|------|--------|
| 1 | **제품 의사결정의 언어가 없다** — 레이아웃, 색상, 타이포, 간격을 "왜 이렇게" 결정하는 기준 자체를 모름. 티켓에 적힌 대로 구현만 해옴 | 지속적 | ★★★★★ |
| 2 | **독립하면 디자인에서 멈춘다** — 사이드 프로젝트나 솔로 제품을 만들려 할 때, 구현은 되는데 UI 의사결정에서 반복적으로 막힘 | 프로젝트마다 | ★★★★★ |
| 3 | **"개발자가 만든 티"가 난다** — 기능은 되는데 왜 어색한지 스스로 진단하지 못함. 결과물에서 신뢰감이 빠져 있음 | 매번 | ★★★★☆ |
| 4 | 디자이너의 피드백("여기 좀 다듬어줘")을 해석하지 못한다 | 조직 내 | ★★★☆☆ |
| 5 | 디자인을 VOD로 배우면 추상적이라 "감각의 영역"으로 남는다 — 동료 피드백 없이는 "내가 맞게 하고 있는지" 확인 불가 | 학습 시도마다 | ★★★★☆ |
| 숨겨진 | **"디자인은 감각의 영역"이라서 내가 배워서 될 게 아니라고 생각함** — 체계적으로 접근 가능하다는 걸 모름 | 고착된 믿음 | ★★★★★ |

숨겨진 페인포인트의 중요성: HN "I'm bad at design" 스레드에서 정확히 이 패턴이 반복됨 — "내가 만든 게 쓰레기 같아서 의욕을 잃는다" → "기존 제품 디자인을 그대로 베꼈더니 동기가 돌아왔다". 이 사람에게 필요한 건 디자인 이론이 아니라 **"이 상황에서 이렇게 하면 됨"이라는 결정 사전**. 정확히 Vibe Dictionary.

### 상승 욕구

**욕망의 핵심: 파이프라인의 부품에서 제품 의사결정자로**

이 사람은 디자이너가 되고 싶은 게 아님. 티켓 바깥에서 제품 전체를 다룰 수 있는 사람이 되고 싶은 것. 빠진 언어(디자인 의사결정)를 채우는 욕망.

- 욕망 강도: ★★★★★ (고용 불안정이 추가 압력 — 부품으로만 남으면 대체됨)
- 트리거: 레이오프, 업무 확대 요구, 사이드 프로젝트에서 디자인 앞 좌절 경험
- 학습 곡선: 코드를 이미 쓸 줄 알기 때문에 VDL 커리큘럼의 기술적 부분은 빠르게 소화. 디자인 결정 프레임워크가 핵심 가치
- 심리적 허들: "디자인 교육에 돈을 쓰는 것"이 개발자 정체성과 충돌할 수 있음 → **"디자인을 배우는 게 아니라 제품 의사결정 언어를 갖추는 것"으로 프레이밍**

### 불안 요소

| 불안 유형 | 구체적 형태 | 강도 |
|----------|------------|------|
| 생존 불안 | "레이오프 당했는데, 혼자서 뭘 만들어서 먹고살 수 있을까?" | ★★★★★ |
| 부품 불안 | "티켓만 처리하는 사람은 결국 AI로 대체되는 거 아닌가?" | ★★★★★ |
| 존재적 불안 | "AI가 코드를 쓰는데, 코딩만 하는 나의 가치는?" | ★★★★☆ |
| 정체성 불안 | "디자인 교육에 돈을 쓰는 게 개발자로서 맞는 투자인가?" | ★★★☆☆ |

VDL의 해소 전략:
- 생존 불안 → "제품 전체를 다룰 수 있는 사람만 독립할 수 있다. 빠진 언어가 디자인 의사결정이다"

- 정체성 불안 → "디자인을 배우는 게 아니라 제품 의사결정의 빠진 절반을 채우는 것" — 개발자 정체성 유지 프레임

### 세 층위 반응

| 층위 | 이 사람에게 어떻게 작동하는가 | 활성 시점 |
|------|------------------------------|----------|
| ① 커리어 상승 | "구현만 하는 개발자는 부품이다. 제품 의사결정까지 하는 개발자는 프로덕트 단위로 수익화된다." | Attract |
| ② 대체 불안 해소 | "AI가 코드를 쓰는 시대에 코딩만 하는 개발자의 가치는?" → "무엇을, 왜 만들어야 하는지를 결정하는 사람은 AI가 대체 못 한다" | Engage + 모듈 1~2 |
| ③ 디자인 본질 회귀 | "감각이라고 생각했던 디자인이 사실은 명명 가능하고 로직으로 분해 가능한 체계였다" → 디자인이 부담에서 무기로 전환됨 | 모듈 2~3 |

③ 도달 가능성: **높음** — "감각 → 로직"이라는 전환이 개발자의 사고방식과 정확히 맞기 때문. "체계로 분해할 수 있다"는 깨달음이 개발자에게는 특히 강렬. 진입 동기가 "배우고 싶다"가 아니라 "멈추고 싶지 않다"이기 때문에, ③에 도달하면 사후적으로 "아, 이게 디자인이었구나"라는 반전이 일어남.

### 구매 행동

| 지표 | 수치 | 근거 |
|------|------|------|
| 리드 전환 | 5~8% | "디자인 교육" 프레임이면 3%, "제품 의사결정 언어" 프레임이면 8%까지 |
| 커뮤니티 전환 | 25~35% | Dictionary를 보고 "이게 결정 기준이구나" 인식되면 관여도 높음 |
| 유료 전환 | 5~7% | 독립 압박이 있는 사람(솔로프리너, 사이드 프로젝트)은 전환 빠름 |
| 코호트 완주율 | 85%+ | 코드 기반이 있어서 기술적 탈락 적음 |

### 커뮤니케이션 가이드

| 상황 | 톤 | 메시지 예시 |

| 졸업 후 증언 유도 | 전환 경험 | "디자이너가 왜 이 간격을 고집하는지 이해하게 됐다. 감각이 아니라 체계였다." |
