## 0. 히어로가 증명해야 하는 것

히어로는 VDL의 전체 명제를 30초 안에 체감시켜야 한다. 브랜딩 3개 문서의 교차 분석:

| 문서 | 히어로에서 전달해야 하는 것 |
|------|------------------------|
| **전략 v2.1 §6.1** | Before/After — "예쁘게 만들어줘" vs 정밀한 디자인 어휘의 차이. 30초 증명 |
| **비주얼 v0.4 §6-7** | Naming Line + 아이소메트릭 분해 + 1px monoline + violet-gray dark + grid 배경 |
| **버벌 v1.0** | "바이브 코딩을 디자인 툴처럼 사용합니다" (태그라인) + 산세리프/모노스페이스 공존 |
| **전략 v2.1 §1.2** | "도구는 이미 있다. 빠진 건 도구를 지휘하는 언어다" |

**핵심 긴장:** 도구를 보여주는 것이 아니라, **디자인 언어가 결과를 바꾼다는 것**을 인터랙션으로 체감시킨다.

## 1. Code Aesthetic 트렌드 리서치

### 1.1 "Technical Mono" — 2026년 핵심 시각 트렌드

Ioana Teleanu(前 Miro AI Lead Product Designer)가 "Aesthetics in the AI Era: Visual + Web Design Trends for 2026" 아티클에서 명명한 트렌드. 개발자 문화와 디자인의 융합에서 탄생한 미학.

**정의:** 모노스페이스 타이포그래피 + 커맨드라인의 단순함 + 하이콘트라스트 레이아웃. 터미널 윈도우와 구형 IBM의 감각을 현대적으로 재해석한 것.

**핵심 특징:**
- 모노스페이스 폰트(JetBrains Mono, OCR-A 등)를 코드뿐 아니라 본문/헤딩에도 사용
- 텍스트 중심, 미니멀 인터페이스, 그리드 기반 구조
- 흑백 또는 green-on-black 컬러 스킴
- ASCII 아트, 커서 아이콘, 와이어프레임 다이어그램을 장식 요소로 활용
- **"기술적 진정성(technical authenticity)"** 시그널링

**문화적 맥락:** "빌더 서브컬처 — VS Code와 Figma를 동시에 사는 사람들"이라는 묘사가 VDL 타겟(캔버스 디자이너, 파이프라인 개발자)과 정확히 일치한다.

**주류화 전망:** 2026년, Technical Mono가 메인스트림 디자인 시스템에 영향을 미치기 시작. 주요 브랜드가 모노스페이스 타입을 미래지향적이면서 날것의 느낌으로 채택. 지나치게 광택 나는 UI에 대한 반작용으로, 커맨드라인의 정신을 현대 디자인에 되돌리는 움직임.

### 1.2 Code Aesthetic의 하위 분류

| 분류 | 특징 | VDL 적합성 | 대표 사이트 |
|------|------|-----------|-----------|
| **Terminal Noir** | 순수 터미널 감각. green-on-black, ASCII, CLI 시뮬레이션 | ⚠️ 과도한 "해커" 느낌. VDL 절제 원칙·온도 원칙(v0.4 원칙 2)과 충돌 | Warp, 해커 포트폴리오 |
| **Technical Mono** | 모노스페이스 + 그리드 + 하이콘트라스트. 코드의 구조적 아름다움 | ✅ VDL의 "산세리프+모노 공존"(v0.4 §4.1)과 정확히 일치 | Monaspace, ASTA, Resend |
| **IDE Aesthetic** | 코드 에디터 UI를 웹 디자인 언어로 차용. syntax highlighting이 비주얼 | ✅ Before/After 프롬프트 시연(전략 v2.1 §6.1)에 최적 | Stripe, AuthZed |
| **Parametric/Generative** | 코드로 생성된 비주얼. 파라미터로 정의 가능한 형태 | ✅✅ VDL 비주얼 원칙 0 "감각도 쪼개면 로직이다"의 직접적 시각화 | A' Design Award 수상작 |

**VDL 최적 조합: Technical Mono + IDE Aesthetic + Parametric**

Terminal Noir의 과도한 해커 감각은 피하되, Technical Mono의 구조적 아름다움 + IDE의 코드 시연 패턴 + Parametric의 "파라미터로 정의 가능한 형태"를 결합하면 VDL 비주얼 가이드와 완벽히 부합한다.

### 1.3 키 비주얼 아이데이션 v0.1 §2 "Code is Beautiful"과의 관계

키 비주얼 아이데이션 v0.1의 세 방향 중 [2] Code is Beautiful은 "말이 코드가 되고, 코드가 디자인이 된다"를 선언한다. Code Aesthetic 리서치는 이 방향의 **시장 검증**이다:

- v0.1 §2.2의 `<V>` 형상 체계 → Technical Mono의 모노스페이스 로고 트렌드와 시대적 공명
- v0.1 §2.3의 "말→코드→디자인 흐름 시각화" → IDE Aesthetic의 코드 시연 패턴으로 구현 가능
- v0.1 §2.4의 "코드 두려움 제거 전략" → Terminal Noir를 피하고 Technical Mono를 택해야 하는 근거
- v0.1 §2.5의 시각 블랙리스트 (Matrix 스타일, 빽빽한 코드 벽 금지) → Terminal Noir 배제와 일치

## 2. 레퍼런스 사이트 분석 (2025.09 ~ 2026.03)

### Tier 1 — VDL Code Aesthetic에 직접 적용 가능

#### REF H-1. Monaspace (GitHub Next)

- **URL:** monaspace.githubnext.com
- **유형:** 모노스페이스 서체 런칭 사이트
- **Code Aesthetic 요소:** 다크 배경 + 5개 모노스페이스 서체 인터랙티브 시연 + texture healing 시각화. 코드 타이포그래피 자체가 히어로
- **VDL 적합도:** ★★★★★
- **VDL 적용 가능성:**
  - "코드 폰트의 아름다움"을 체험으로 증명하는 구조가 VDL의 "코드 = 소재"(v0.1 §4.3)와 일치
  - 타이포그래피 파라미터(웨이트, 리가처, 텍스처 힐링)의 인터랙티브 조작 → VDL의 "파라미터로 정의 가능"(v0.4 원칙 0)과 공명
  - 히어로에서 디자인 키워드가 모노스페이스로 표시되고, 호버 시 그 키워드의 의미가 시각적으로 변환되는 인터랙션의 근거

#### REF H-2. ASTA Design System

- **URL:** anton.io/asta
- **유형:** 모노스페이스 그리드 기반 UI 프레임워크
- **Code Aesthetic 요소:** JetBrains Mono 기반 캐릭터 그리드 정렬. 12컬럼 레이아웃이 모노스페이스 그리드 유지. 다크 모드 내장
- **VDL 적합도:** ★★★★
- **VDL 적용 가능성:**
  - "체계가 그래픽이다"(v0.4 원칙 3)의 직접적 구현. 모노스페이스 그리드 자체가 시각적 리듬을 만든다
  - VDL의 8px Grid 배경(v0.4 §5.2)과 캐릭터 그리드의 결합 가능성
  - 히어로에서 Grid 위에 모노스페이스 텍스트가 캐릭터 단위로 정렬되는 "코드의 구조적 아름다움" 표현

#### REF H-3. Resend

- **URL:** resend.com
- **유형:** 이메일 API (개발자 도구)
- **Code Aesthetic 요소:** 다크 + 모노스페이스가 히어로 자체. 코드 블록이 비주얼의 중심. 극한 미니멀
- **VDL 적합도:** ★★★★★
- **VDL 적용 가능성:**
  - "코드가 장식이 아니라 작동하는 설계 명세"(v0.1 §2.5 권장 사항)의 완벽한 사례
  - 히어로에서 코드 3-5줄이 곧 비주얼인 구조 → VDL Before/After 프롬프트 시연에 직접 차용
  - 유채색은 구문 하이라이팅에서만 허용(v0.4 §3.1)과 Resend의 접근이 일치

#### REF H-4. Factory AI

- **URL:** factory.ai
- **유형:** AI 에이전트 개발 도구
- **Code Aesthetic 요소:** 다크 + 모노스페이스 + 에이전트 코딩 시각화. "코드가 스스로 작동하는" 인터랙션

- **VDL 적용 가능성:**

#### REF H-5. Terminal Industries (Awwwards SOTM 2025.10)

- **URL:** Awwwards Site of the Month, 2025년 10월
- **유형:** 산업 물류 회사

### Tier 1-B 종합 분석

**AI-Native Agent Aesthetic의 공통 패턴:**
1. **다크 배경 + 하이콘트라스트 텍스트** — 예외 없음. 100% 다크
2. **프롬프트/자연어 입력이 히어로 중앙** — Cursor, Devin, Replit, v0, Bolt 모두
3. **실행 과정의 실시간 시각화** — 코드가 생성되는/실행되는 장면 자체가 비주얼
4. **모노스페이스 + 산세리프 듀얼** — 코드 = 모노, UI/카피 = 산세리프
5. **미니멀 색상** — 유채색은 syntax highlighting 또는 단일 액센트에만 허용

**VDL과의 교차점:**
- 패턴 1~5 모두 VDL 비주얼 가이드(v0.4)와 구조적으로 일치
- 특히 패턴 2(프롬프트 중심 히어로)와 패턴 3(실행 과정 시각화)이 Direction B와 직접 연결

**VDL 차별화 지점:**
- 위 사이트들은 "도구의 성능"을 증명. VDL은 **"어휘의 힘"**을 증명 — 같은 시각 문법에서 메시지만 다름
- v0/Bolt/Replit이 "아무 프롬프트나 넣어도 된다"를 보여준다면, VDL은 "정밀한 프롬프트가 다른 결과를 만든다"를 보여줌
- Poolside/Magic의 연구소 미니멀 ↔ Replit/Bolt의 접근성 — VDL은 이 스펙트럼의 중간("Labs" 정체성 + 교육 접근성)

### Tier 1-C — Design / Creative 에이전시 & 디자인 도구

### Tier 1-C 종합 분석

**Design / Creative 에이전시의 공통 패턴:**
1. **타이포그래피가 히어로 그 자체** — 일러스트/사진이 아니라 글자가 비주얼 (Obys, Unseen, Superlist)
2. **모션이 체계적** — 랜덤 애니메이션이 아니라 타이밍/이징에 의도가 보임
3. **다크 배경에서의 위계 구축** — 화이트 + 단일 액센트로 정보 위계를 명확히
4. **도구/플랫폼의 경우 "데모가 히어로"** — Framer, Tokens Studio 모두 설명이 아니라 체험

**디자인 교육 경쟁자 분석:**
| | Shift Nudge | SuperHi | VDL |
|--|-------------|---------|-----|
| 핵심 명제 | 시각 디자인 스킬 | 디자이너를 위한 코딩 | 디자인 어휘 = 프롬프트 능력 |
| 톤 | 다크 + 전문가 | 밝음 + 친근 | 다크 + Labs 연구 감각 |
| 차별화 | 크래프트 | 접근성 | 어휘 체계 |

**VDL 히어로에 대한 시사점:**
- Tier 1-B(개발 도구)는 "프롬프트 → 결과"를 증명하고, Tier 1-C(디자인)는 "체계 → 아름다움"을 증명
- VDL 히어로는 두 축을 결합해야 함: **"정밀한 디자인 어휘(체계) → 정밀한 프롬프트 → 정밀한 결과(아름다움)"**
- Tokens Studio가 "디자인 토큰 = 로직"이라면, VDL은 "디자인 어휘 = 로직" — 추상화 수준만 다름
- Obys/Unseen의 타이포 중심 히어로 + Cursor/v0의 프롬프트 중심 히어로를 결합하면 Direction A+B의 시각적 근거가 강화됨

## 3. 히어로 인터랙티브 요소 아이데이션 (5가지 방향)

### Direction A: "Naming Line이 살아 움직이는 UI 해부"

사용자가 히어로의 UI 목업 위에 마우스를 올리면 Naming Line이 그 요소에서 뻗어 나오며 디자인 키워드(`Bento Grid`, `8px grid`, `semantic color token`)가 모노스페이스로 표시되는 인터랙션.

- **핵심 체험:** "보이는 것에 이름을 붙이는 행위"를 직접 경험
- **기술:** SVG path animation + mouse tracking + staggered reveal
- **비주얼 원칙 부합:** Naming Line(v0.4 §6.1), Border Highlight(v0.4 §6.3), 원칙 4(보이지 않는 것을 보이게)
- **Code Aesthetic 분류:** Technical Mono + Parametric
- **레퍼런스:** Linear.app(monoline 연결선), Awwwards Hovers & Cursors Collection, Marga Navarro(커서 마스크 리빌)

### Direction B: "Before/After 프롬프트 라이브 시연"

좌우 스플릿 화면. 왼쪽에 "예쁘게 만들어줘"가 타이핑되며 모호한 결과가 렌더링되고, 오른쪽에 "Bento Grid + Glassmorphism + semantic color token"이 타이핑되며 정밀한 결과가 나오는 애니메이션.

- **핵심 체험:** "정밀하게 말하면 정밀하게 나온다" (The Vibe Standard)를 30초 안에 증명
- **기술:** 시뮬레이션 코드 에디터 + 단계별 렌더링 애니메이션
- **비주얼 원칙 부합:** 산세리프+모노 공존(v0.4 §4.1), Before/After(v0.4 §7.2), 원칙 3(체계가 그래픽이다)
- **Code Aesthetic 분류:** IDE Aesthetic
- **레퍼런스:** Stripe(코드 에디터 + 쉘 시뮬레이션), Resend(코드 = 히어로), frontendfyi/rebuilding-linear.app(기술 스택 동일)

### Direction C: "아이소메트릭 레이어 분해"

UI 컴포넌트가 평면에서 시작해 스크롤/클릭에 따라 30° 아이소메트릭으로 z축 분해되며 각 레이어(데이터 → 컴포넌트 → 스타일 → 레이아웃)에 Naming Line이 부착되는 3D 전환.

- **핵심 체험:** "감각도 쪼개면 로직이다" (Philosophy ③)를 시각적으로 증명
- **기술:** CSS 3D transform + SVG Naming Line + scroll-triggered animation
- **비주얼 원칙 부합:** 아이소메트릭 분해(v0.4 §7.1), 원칙 0(감각도 쪼개면 로직이다)
- **Code Aesthetic 분류:** Parametric
- **레퍼런스:** iFixit/Xreart(v0.1 REF 3-C), Awwwards Isometric 3D Navigation

### Direction D: "등가 다이어그램 인터랙션"

`[디자인 키워드]` — `[토큰]` — `[코드]` 3컬럼이 Circuitry 연결선으로 이어지며, 하나를 호버하면 대응하는 나머지 두 컬럼이 하이라이트되는 인터랙션. "디자인 키워드 ↔ 토큰 ↔ 코드" 등가 구조를 체험.

- **핵심 체험:** "구현은 언어를 따른다" (Philosophy ①) + Dictionary의 물적 증거
- **기술:** SVG 연결선 + CSS transition + 3컬럼 반응형
- **비주얼 원칙 부합:** Circuitry 연결선(v0.4 §6.2), 등가 다이어그램(v0.4 §7.2), 원칙 4
- **Code Aesthetic 분류:** Technical Mono + IDE Aesthetic
- **레퍼런스:** Liveblocks.io(연결 관계 시각화), ASTA Design System(모노스페이스 그리드)

### Direction E: "SVG Line Drawing + 타이포그래피 빌드업"

히어로 로딩 시 1px monoline이 grid 위에서 자가 드로잉되며 UI 윤곽을 그리고, 그 위에 Naming Line이 뻗으며 히어로 카피가 등장. 정적 이미지가 아니라 **만들어지는 과정 자체**가 히어로.

- **핵심 체험:** "Design As Build" — 설계 과정이 곧 결과물이 되는 순간
- **기술:** SVG stroke-dashoffset animation + 타이밍 시퀀스
- **비주얼 원칙 부합:** Monoline 공통 파라미터(v0.4 §5.3), Grid 배경(v0.4 §5.2), 원칙 0
- **Code Aesthetic 분류:** Parametric + Technical Mono
- **레퍼런스:** Awwwards Dark Minimal Hero Section(Hardik Bhansali), Awwwards Hero Section Animation

## 4. VDL 적합성 검증 — 위험 요소와 대응

### 4.1 Code Aesthetic이 VDL 원칙과 부합하는 지점

| VDL 원칙 | Code Aesthetic 대응 |
|----------|-------------------|
| 원칙 0: 감각도 쪼개면 로직이다 | 코드 = 로직의 가장 순수한 표현. 코드가 곧 비주얼이 되는 것 자체가 이 원칙의 증명 |
| 원칙 1: 제거가 크래프트다 | Technical Mono는 장식 요소를 자연스럽게 배제. 모노스페이스 그리드만으로 시각 체계 |
| 원칙 3: 체계가 그래픽이다 | Technical Mono에서 모노스페이스 그리드 자체가 그래픽 역할 |
| §4.1: 산세리프+모노 공존 | Code Aesthetic의 핵심이 바로 이 공존 |
| §2.1 블랙리스트 | Code Aesthetic은 그래디언트, 블러, 글라스모피즘을 자연스럽게 배제 |
| §3.1: 유채색은 코드 구문에서만 | IDE Aesthetic의 syntax highlighting = VDL의 유일한 유채색 허용 |

### 4.2 위험 요소

| 리스크 | 근거 | 대응 |
|--------|------|------|
| "교육 업체"가 아니라 "개발자 도구"처럼 보일 수 있음 | Terminal Industries가 물류 회사인데 터미널 미학 → 업종 오인 가능성 | Naming Line으로 디자인 키워드를 전면에 → "코드의 언어"가 아니라 "디자인의 언어"임을 명시 |
| 디자이너 타겟(70%)이 코드 미학에 위축될 수 있음 | v0.1 §2.5 블랙리스트: Matrix 스타일, 빽빽한 코드 벽 = 두려움 유발 | Before/After로 "이 코드가 아니라 이 어휘가 결과를 바꾼다"를 증명. 코드를 3-5줄만 노출 |
| VDL 퍼스널리티 "소재의 온도"(원칙 2)와 충돌 | Code Aesthetic은 기본적으로 차가운 감각 | violet-tinted grayscale(v0.4 §3.1)로 미세한 온기 부여. 순수 #000 금지 |
| "바이브 코딩 교육 = 강의 팔이" 연상 (전략 v2.1 §8.2) | Code Aesthetic의 정밀함이 "Labs"의 연구 감각과 일치하면 이 리스크를 상쇄 | 히어로에서 "배우세요" 톤이 아니라 "연구합니다" 톤 유지 |
| Technical Mono가 트렌드로 소비되고 1년 뒤 유효하지 않을 수 있음 | v0.4 §10 #6: "1년 뒤에도 유효한가?" | VDL의 Code Aesthetic은 트렌드 추종이 아니라 브랜드 정체성과의 구조적 일치. 모노스페이스+산세리프 공존은 v0.4에서 이미 확정된 원칙 |

## 5. 추천 조합과 우선순위

### 5.1 VDL 블랙리스트 검증

다섯 방향 모두 비주얼 블랙리스트(v0.4 §2.1)를 위반하지 않음:
- ❌ 그래디언트 배경 / 블러 글로우 → 사용 안 함
- ❌ 글라스모피즘 → 사용 안 함
- ❌ 뉴럴 네트워크 / 로봇 이미지 → 사용 안 함
- ❌ 캐릭터 / 마스코트 → 사용 안 함
- ❌ 추상적 블롭 / 유기적 형태 → 사용 안 함
- ❌ 스톡 사진 / 장식적 3D → 사용 안 함

### 5.2 추천 조합

**1순위: Direction A + B (Naming Line 호버 + Before/After 프롬프트 시연)**

> Naming Line 호버 인터랙션(A)을 배경으로, Before/After 프롬프트 시연(B)을 중앙에 배치. "보이는 것에 이름을 붙이는 행위"가 곧 "정밀한 프롬프트"로 연결되는 서사.

- **전략 적합:** 30초 증명(전략 §6.1), The Vibe Standard(전략 §1.5.2)
- **비주얼 적합:** Naming Line(v0.4 §6.1) + Before/After(v0.4 §7.2)
- **Code Aesthetic:** Technical Mono + IDE Aesthetic
- **핵심 레퍼런스:** Resend(코드 = 히어로) + Linear(monoline 연결선) + Stripe(코드 에디터 시뮬레이션)

**2순위: Direction E + A (SVG Line Drawing 빌드업 + Naming Line 호버)**

> SVG line drawing 빌드업(E)으로 시작해 UI가 완성되면 Naming Line 호버(A)가 활성화. "설계가 곧 구현" → "모든 요소에 이름이 있다"로 이어지는 2단계 경험.

- **전략 적합:** Design As Build(전략 §1.5.2)
- **비주얼 적합:** Monoline(v0.4 §5.3) + Grid(v0.4 §5.2) + Naming Line(v0.4 §6.1)
- **Code Aesthetic:** Parametric + Technical Mono
- **핵심 레퍼런스:** Monaspace(타이포 = 히어로) + ASTA(모노스페이스 그리드)

**3순위: Direction C + D (아이소메트릭 분해 + 등가 다이어그램)**

> 아이소메트릭 분해(C)로 구조를 노출한 뒤, 등가 다이어그램(D)으로 키워드-토큰-코드 관계를 보여주는 스크롤 시퀀스. 스크롤 기반이라 히어로 하단~첫 섹션 연결에 적합.

- **전략 적합:** Philosophy ①③, Dictionary의 물적 증거
- **비주얼 적합:** 아이소메트릭 분해(v0.4 §7.1) + Circuitry(v0.4 §6.2) + 등가 다이어그램(v0.4 §7.2)
- **Code Aesthetic:** Parametric
- **핵심 레퍼런스:** Liveblocks(연결 관계) + Awwwards Isometric
