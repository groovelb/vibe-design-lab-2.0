# VDL 버벌 브랜딩 — 롱비전 / 숏비전 분리 제안

> 상위 문서: VDL 버벌 브랜드 아이덴티티 v1.0
> 기반 데이터: VDSK 수업 신청 설문 104명 응답 분석
> 작성일: 2026-03-09

---

## 0. 배경: 왜 분리가 필요한가

### 현재 상태

VDL 버벌 브랜딩은 전부 **롱비전 레이어**에서 작동하고 있다.
"Design at the Speed of Thought", "도구를 지휘하는 언어", "의도의 정밀도" — 전부 VDL이 믿는 진실이고, 코호트를 마친 사람이 도달하는 상태다.

### 설문에서 드러난 갭

설문 104명 중 대다수는 아직 그 언어를 체감할 수 없는 상태다.

| 유저의 실제 언어 | 현재 브랜딩 언어 | 갭 |
|---|---|---|
| "코드를 모르니까 막히면 속수무책이다" | "의도의 정밀도가 결과를 결정한다" | 유저는 "의도의 정밀도"를 아직 경험한 적 없다 |
| "AI 티가 난다" | "System Over Drawing" | 용어 자체가 유저의 현재 어휘에 없다 |
| "코딩을 모르면 안 되는 거 아닌가" | "코드를 배우면 디자인을 더 정확하게 말할 수 있다" | 유저의 불안을 직접 해소하지 못한다 |

### 분리 원칙

숏비전은 유저가 **지금 서 있는 자리에서 출발하는 약속**이고, 롱비전은 **VDL이 데려가려는 곳**이다.

이 두 개가 분리되어야 퍼널 단계마다 다른 카피를 쓸 수 있다.

### 1.4 디자인 시스템 수요 (1+2순위 합산)

"디자인 시스템 관리 및 코드 기반 디자인 옵스 능력"을 1순위 또는 2순위로 고른 사람: **70명 (67%)**. 전 직군에서 가장 넓은 수요.

## 2. 갭 분석: 설문 언어 vs 버벌 브랜딩

### 2.1 캔버스 디자이너 페인포인트 갭

현재 페인포인트: **"내 디자인인데 완성은 남의 몫이다"** — 디자이너-개발자 협업 구조의 한계를 전제.

설문 속 디자이너 실제 고통: 이미 바이브 코딩을 시도하거나 시도하려는 상태이고, 고통은 **"혼자 하고 있는데 막히면 어디서부터 고쳐야 할지 모르겠다"**에 가깝다.

유저 원문:
- "개발 지식이 없으니 오류가 발생할 때 어떻게 해결할 수 있을지"
- "개발 프로세스를 잘 몰라 원하는게 안나오면 어디서부터 어떻게 해결해야할지 아예 감이 없음"
- "코드를 이해하지 못한 채 복붙으로 끝날까봐, 막연한 두려움"
- "막무가내로 프롬프트를 쓰고 구현하다보니 오히려 리팩토링에 시간이 훨씬 걸리는 문제"

### 2.2 파이프라인 개발자 전환 메시지 갭

현재 전환 메시지: **"제품 의사결정의 빠진 절반을 채우는 것"** — 맞는 말이지만 추상적.

개발자 원문:
- "AI 티가 많이 나는 획일적인 디자인만 나오는 점이 가장 고민"
- "진짜 디자이너가 한 것처럼 예쁘게 디자인 하고 싶은 맘이 큽니다"
- "production 수준의 디자인 만들기"
- "디자인에 대해서 잘 몰라서 좋은 디자인에 대한 기준조차 없음"

### 2.3 "독립" 동기 위치 갭

유저의 1/3(32%)이 "개인 사업/IT 서비스 구축"이 동기. 현재 버벌 브랜딩에서 "독립"은 약속의 시간축 **장기**(형태를 스스로 고를 수 있는 상태)에 배치되어 있지만, 이 사람들에게는 **단기 동기**다.

### 2.4 Myth-Busting 갭

설문에서 가장 큰 통념 **"코딩을 알아야 바이브 코딩을 잘할 수 있다"**(35명)가 현재 편견 깨기 3개에 빠져 있다.

### 2.5 "디자인 시스템" 용어 갭

1+2순위 합산 67%가 "디자인 시스템"을 기대. 버벌 브랜딩에서 "디자인 시스템"은 Dictionary, 디자인 언어, 어휘 체계로 치환되어 있는데, 유저가 기대하는 범위(토큰, 컴포넌트, 코드 기반 관리, Figma 동기화)를 직접 가리키는 단어가 노출되지 않는다.

## 3. 롱비전 (Long Vision) — VDL이 데려가려는 곳

코호트를 마치고, Dictionary를 쓰면서 체감하는 상태. 학습자가 스스로 도착하는 곳.

### 3.1 브랜드 가치

**생각과 구현 사이의 격차를 없앤다**

### 3.2 고정 문구 (기존 유지)

| 구분 | 문구 | 사용처 |
|---|---|---|
| 슬로건 | Design at the Speed of Thought | 커뮤니티, 교육 내부, 졸업생 |
| 태그라인 | AI가 이해하는 디자인 언어를 연구합니다 | About 페이지, 프로필 |
| 한 문장 정의 | 디자인하듯이 쓰는 바이브 코딩, 생각과 구현의 격차를 줄이는 디자인 언어를 연구합니다 | 소개, About |
| 미션 | 생각하는 사람이 직접 만들 수 있게 합니다 | 미션 페이지, 내부 |

### 3.3 적용 범위

"이미 VDL을 아는 사람"이 보는 곳: 커뮤니티 내부, 졸업생 증언, About 페이지, 미션 섹션, 코호트 내부.

### 3.4 페르소나별 롱비전 문구 (기존 유지)

**캔버스 디자이너:**

| 용도 | 문구 |
|---|---|
| 페인포인트 | 내 디자인인데 완성은 남의 몫이다 |
| 전환 메시지 | 디자인 의도를 끝까지 통제하는 수단을 갖추는 것 |
| 도달 상태 | 내 의도가 그대로 구현된다 |

**파이프라인 개발자:**

| 용도 | 문구 |
|---|---|
| 페인포인트 | 왜 어색한지 모르겠다 |
| 전환 메시지 | 제품 의사결정의 빠진 절반을 채우는 것 |
| 도달 상태 | 기준이 생기면 판단이 된다 |

**샌드위치 PM:**

| 용도 | 문구 |
|---|---|
| 페인포인트 | 말로 설득해야 하는데 보여줄 게 없다 |
| 전환 메시지 | 검증을 직접 할 수 있는 PM이 되는 것 |
| 도달 상태 | 설득이 아니라 증명으로 일한다 |

## 4. 숏비전 (Short Vision) — 유저가 지금 서 있는 자리

설문 데이터에서 추출한 유저의 실제 고통에서 출발. Attract 단계에서 유저를 VDL 세계로 데려오는 입구.

### 4.1 브랜드 가치 (숏비전)

**바이브 코딩의 결과를 통제하는 기준을 만든다**

### 4.2 핵심 통찰

유저가 공통으로 느끼는 하나의 감정: **통제력의 부재**.

| 유저의 언어 | 공통 감정 |
|---|---|
| "코드를 모르니까 막히면 속수무책이다" (35명) | 통제 불가 |
| "AI가 만든 티가 난다 / 의도대로 안 나온다" (13명) | 통제 불가 |
| "혼자서 끝까지 해낼 수 있을지 모르겠다" (9명) | 통제 불가 |

VDL의 답: 코드 지식이 아니라 **디자인 기준**이 통제력을 만든다.

### 4.3 숏비전 태그라인 후보

```
A. 바이브 코딩, 감으로 하지 않습니다
B. AI 결과물을 통제하는 디자인 기준
C. 되는 대로가 아니라 의도대로
```

### 4.4 적용 범위

"VDL을 아직 모르는 사람"이 처음 보는 곳: SNS 첫 접점, 광고, 코호트 모집 페이지 히어로, Threads 프로필.

### 4.5 페르소나별 숏비전 문구 (신규)

**캔버스 디자이너:**

| 용도 | 문구 | 근거 |
|---|---|---|
| 페인포인트 | 혼자 만들고 있는데, 막히면 어디서부터 고쳐야 할지 모르겠다 | 설문 35명 "개발 지식/코드 이해 부족" |
| 전환 메시지 | 오류가 나도 원인을 찾을 수 있는 기준을 갖는 것 | "어디서부터 해결해야할지 아예 감이 없음" 해소 |
| 도달 상태 | "되는 대로" 만드는 단계를 벗어난다 | "막무가내 프롬프트 → 리팩토링 반복" 탈출 |
| 정체성 보호 | 코딩을 배우는 게 아니라, 내 디자인의 기준을 코드로 전달하는 것 | "코드를 이해 못한 채 복붙으로 끝날까봐" 해소 |

**파이프라인 개발자:**

| 용도 | 문구 | 근거 |
|---|---|---|
| 페인포인트 | 만들 수는 있는데 AI 티가 난다 | 설문 "AI 티가 많이 나는 획일적인 디자인" |
| 전환 메시지 | "이게 왜 어색한지"를 설명할 수 있는 기준을 갖는 것 | "좋은 디자인에 대한 기준조차 없음" |
| 도달 상태 | 디자이너 없이도 production 수준이 나온다 | "production 수준의 디자인 만들기" |
| 정체성 보호 | 디자인을 배우는 게 아니라, AI 결과물의 어색함을 잡는 기준을 갖는 것 | "서비스를 만들면서 디자인으로 스트레스 받고 싶지 않아요" |

**샌드위치 PM:**

| 용도 | 문구 | 근거 |
|---|---|---|
| 페인포인트 | 기획은 했는데 검증할 수단이 없다 | 설문 PM 7/10명이 "개인 사업/사이드 프로젝트" 동기 |
| 전환 메시지 | 기획서를 직접 돌아가는 결과물로 만드는 것 | "검증을 직접" + 독립 제작 동기 |
| 도달 상태 | 아이디어 단계에서 프로토타입까지 혼자 간다 | 사내 설득보다 독립 제작이 주 동기 |
| 정체성 보호 | 개발자가 되라는 게 아니라, 기획을 돌아가는 형태로 만드는 것 | "디자인도 개발도 전문가가 아닌 내가 잘할 수 있을지" |

## 5. 히어로 카피 제안

### 5.1 롱비전 히어로 (기존 유지 — About, 미션 페이지)

```
디자인하듯이 쓰는 바이브 코딩
— AI가 이해하는 디자인 언어를 연구합니다
```

### 5.2 숏비전 히어로 (신규 — 코호트 모집, 광고, SNS 첫 접점)

```
후보 A:
바이브 코딩, 되는 대로 만들고 있다면
— 결과가 달라지는 건 도구가 아니라 기준입니다

후보 B:
감으로 만드는 바이브 코딩을 졸업합니다
— 디자인 기준이 있으면 AI 결과를 통제할 수 있습니다

후보 C:
"왜 안 되는지 모르겠다"에서 벗어나는 법
— 코딩 지식이 아니라 디자인 기준이 필요합니다
```

후보 C는 설문 최대 불안("코드를 몰라서 막히면 속수무책")을 직접 건드리면서, VDL의 답("필요한 건 코드가 아니라 기준")으로 전환한다.

## 6. Myth-Busting 추가 제안

기존 편견 3개에 1개 추가.

### 편견 4: 바이브 코딩을 잘하려면 코딩을 알아야 한다

근거: Philosophy ① 구현은 언어를 따른다

| 통념 | 개발 지식이 있어야 AI에게 정확한 지시를 할 수 있다 |
|---|---|
| 반전 | AI가 필요로 하는 건 코드 문법이 아니라 설계 기준이다. 간격을 몇 px로 할지, 상태를 몇 가지로 나눌지, 색을 어떤 위계로 쓸지 — 이건 디자인 결정이다 |
| 핵심 문구 | 바이브 코딩의 병목은 코딩 지식이 아니라, 의도를 구체적으로 정의하는 기준이다 |
| 서브 문구 | "에러가 나면 어떡하지"보다 먼저 답해야 할 질문은 "내가 뭘 만들려는 건지 정확히 정의했는가"다 |

설문 근거: 35명(걱정 작성자의 46%)이 "개발 지식 부족"을 최대 걱정으로 꼽았다. 이 통념을 직접 해소하지 않으면 Attract 단계에서 이탈한다.

## 7. 추가 워싱 포인트

### 7.1 "디자인 시스템" 브릿지 문구

유저가 쓰는 단어("디자인 시스템")와 VDL이 쓰는 단어("디자인 언어 체계") 사이 번역이 필요.

제안: 코호트 소개, Dictionary 소개에서 한 번은 "디자인 시스템"을 유저 언어 그대로 노출하되, VDL 프레이밍으로 연결.

```
기존: 200+ 디자인 키워드. 프롬프트에 넣으면 결과가 달라지는 표준 어휘 체계.
제안: 200+ 디자인 키워드. 디자인 시스템의 기준을 AI가 이해하는 언어로 번역한 체계.
```

### 7.2 "독립" 동기 Attract 노출

유저의 32%가 "개인 사업/독립 서비스"가 동기인데, 현재 히어로와 Attract 카피에 이 맥락이 빠져 있다.

제안: 포지셔닝 대비 문구에 1줄 추가.

| VDL ≠ | VDL = |
|---|---|
| (기존) 직업을 바꿔라 | 본업의 반경을 AI로 넓혀라 |
| **(추가) 취업을 위한 교육** | **만들 수 있는 사람이 되는 교육** |

## 8. 두 레이어의 전환 구조

```
숏비전 (Attract)                          롱비전 (Delight / Advocate)
────────────────────────────────────────────────────────────────────

유저의 언어                           →   VDL의 언어
"막히면 속수무책이다"                   →   "의도의 정밀도가 결과를 결정한다"
"AI 티가 난다"                        →   "System Over Drawing"
"코드를 모르면 안 되는 거 아닌가"        →   "필요한 건 코드가 아니라 디자인 언어다"

통제의 약속                           →   언어 체계의 약속
바이브 코딩의 결과를 통제하는 기준       →   생각과 구현 사이의 격차를 없앤다
되는 대로가 아니라 의도대로             →   Design at the Speed of Thought
```

### 전환 시점

| 단계 | 활성 레이어 | 카피 추상도 |
|---|---|---|
| 발견 (Attract) | **숏비전** | 유저 언어. 고통 직접 묘사 |
| 관여 (Engage) | 숏→롱 전환 | Before/After로 결과 차이를 보여주며 VDL 용어 도입 |
| 만족 (Delight) | **롱비전** | VDL 언어. 학습자가 체감으로 이해하는 단계 |
| 옹호 (Advocate) | 롱비전 | 졸업생이 자기 언어로 롱비전을 말하기 시작 |

## 9. 채널별 레이어 매핑

| 채널 | 레이어 | 근거 |
|---|---|---|
| Threads/X (숏폼) | **숏비전** | 첫 접점. 유저 고통의 언어로 스크롤을 멈춰야 한다 |
| 코호트 모집 페이지 히어로 | **숏비전** | 전환 결정 직전. 불안 해소가 핵심 |
| Before/After 콘텐츠 | 숏→롱 **전환** | 숏비전 문제 제시 → 롱비전 해결 시연 |
| Dictionary 소개 | 숏→롱 **전환** | 유저 단어("디자인 시스템") → VDL 단어("표준 어휘 체계") |
| About / 미션 페이지 | **롱비전** | 이미 관심을 가진 사람이 보는 곳 |
| 코호트 내부 콘텐츠 | **롱비전** | 학습자가 VDL 언어를 체화하는 과정 |
| 졸업생 증언 | **롱비전** → 다음 사람의 숏비전 | 플라이휠: ③ 증언이 새 Attract 자산 |
