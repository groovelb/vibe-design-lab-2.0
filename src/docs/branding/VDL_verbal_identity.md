### 1.2 Vibe Philosophy

VDL이 참이라고 믿는 세 가지 신념. 슬로건과 태그라인이 **무엇**을 약속하는지 말한다면, Philosophy는 그 약속이 **왜** 가능한지를 말한다. 아래 가치 제안(1.3)은 이 신념에서 파생된다.

| # | 신념 | 영역 | 한 줄 설명 |
|---|------|------|-----------|
| ① | 구현은 언어를 따른다 | 구현 | 결과물의 품질을 결정하는 건 도구의 성능이 아니라 언어의 정밀도다 |
| ② | 언어는 경험을 함축한다 | 언어 | 하나의 디자인 키워드에는 수십 개의 설계 결정이 압축되어 있다 |
| ③ | 감각도 쪼개면 로직이다 | 디자인 | 창의적이라고 불리는 감성 영역도 분해하면 명명 가능한 결정들의 조합이다 |

**파생 관계:**
- ① 구현은 언어를 따른다 → Design As Build (설계가 곧 구현이다)
- ② 언어는 경험을 함축한다 → The Vibe Standard (정확하게 말하면 정확하게 나온다)
- ③ 감각도 쪼개면 로직이다 → System Over Drawing (기준이 먼저다)

### 1.3 가치 제안 문구

| 가치 제안 | 한 줄 설명 | 숏 버전 |
|----------|-----------|---------|
| System Over Drawing | 결과물보다 기준을 먼저 설계합니다 | 기준이 먼저다 |
| The Vibe Standard | AI가 알아듣는 표준 디자인 언어 | 정확하게 말하면 정확하게 나온다 |
| Design As Build | 설계를 제대로 했다면 구현을 따로 설명할 필요가 없습니다 | 설계가 곧 구현이다 |

인과 흐름: 기준을 먼저 설계하고 → 표준 언어로 말하면 → 설계가 곧 구현이 된다. 이 인과가 작동하는 근거가 Vibe Philosophy(1.2)다.

### 1.5 약속의 시간축 문구

| 시간축 | 핵심 문구 | 서브 카피 |
|-------|----------|----------|
| 단기 (스킬) | "거의 맞는" 결과가 "의도대로"로 바뀐다 | Figma 안에서만 가능했던 작업이 코드, 동작, 배포까지 확장된다 |
| 중기 (커리어) | 지원할 수 있는 직무가 넓어진다 | 디자인 엔지니어, 프로덕트 엔지니어. 팀 안에서 역할 범위가 달라진다 |

### 1.6 포지셔닝 대비 문구

| VDL ≠ | VDL = |
|-------|-------|
| 코드를 몰라도 만들 수 있다 | 코드를 배우면 디자인을 더 정확하게 말할 수 있다 |
| 도구 사용법을 가르치는 곳 | 도구를 지휘하는 언어를 연구하는 곳 |
| 누구나 할 수 있다 | 잘하려면 디자인 언어가 필요하다 |
| 빠르게 만드는 사람 | 자기 판단으로 결과를 내는 사람 |
| 강의를 파는 교육 업체 | 연구하고, 연구 결과로 빌더를 키우는 곳 |
| 직업을 바꿔라 | 본업의 반경을 AI로 넓혀라 |
| 바이브 코딩을 쉽게 | 바이브 코딩의 품질 기준을 |

### 1.7 편견 깨기 (Myth-Busting)

콘텐츠의 각 단락에서 독자의 사고를 반전시키는 장치. 통념을 먼저 꺼내고, 뒤집는다. 각 편견의 근거가 되는 Vibe Philosophy를 함께 표기한다.

**편견 1: 도구 성능이 올라가면 결과도 좋아질 거다** — 근거: Philosophy ① 구현은 언어를 따른다

| 통념 | 더 좋은 모델, 더 좋은 서비스가 나오면 결과가 달라진다 |
|------|------|
| 반전 | 도구는 이미 충분하다. 결과를 결정하는 건 도구의 성능이 아니라 의도의 정밀도다 |
| 핵심 문구 | 도구가 바뀌어도 정밀하지 않은 의도는 정밀하지 않은 결과를 만든다 |
| 서브 문구 | 기다리는 건 더 좋은 도구가 아니라, 의도를 일관되게 전달하는 체계다 |

**편견 2: 디자인은 구현 다음에 다듬는 것이다** — 근거: Philosophy ③ 감각도 쪼개면 로직이다

| 통념 | 일단 만들고, 디자인은 나중에 입히면 된다 |
|------|------|
| 반전 | 디자인은 마지막에 입히는 것이 아니라, 설계와 구현을 동시에 결정하는 언어다 |
| 핵심 문구 | 디자인이 다루는 범위를 알면, 설계가 곧 구현의 명세가 된다 |
| 서브 문구 | "간격을 8px 단위로 쓸지, 색상 토큰을 어떤 위계로 나눌지"는 디자인 결정이면서 구현 설계다. 디자인은 기준을 정의하는 언어이고, 기준이 정의되면 구현이 따라온다 |

**편견 3: AI 시대에는 새로운 스킬을 빨리 익히는 게 중요하다** — 근거: Philosophy ② 언어는 경험을 함축한다

| 통념 | 새 도구, 새 스킬을 빨리 배워야 살아남는다 |
|------|------|
| 반전 | 스킬보다 먼저 바뀌어야 하는 건, 의도와 결과가 불일치하는 구조를 받아들이는 습관이다 |
| 핵심 문구 | 피드백을 기다리고, 해석을 맡기고, 결과를 넘기는 방식이 반복될수록 전문성과 결과물 사이의 격차가 벌어진다. 이 격차를 좁히는 훈련이 스킬보다 먼저다 |
| 서브 문구 | 의도와 결과의 격차가 줄어들면 스킬이 달라지고, 커리어가 달라지고, 일하는 방식을 직접 고를 수 있게 된다 |

## 2. 톤 컨벤션

### 2.1 보이스 규칙

VDL의 보이스는 퍼스널리티 5축에서 파생된다:

| 축 | 글쓰기 규칙 | 예시 |
|---|-----------|------|
| 주도 | 단정적으로 말한다. "~일 수 있습니다"보다 "~입니다" | ❌ 도움이 될 수 있습니다 → ✅ 달라집니다 |
| 정밀 | 추상적 형용사 대신 구체적 키워드·수치를 쓴다 | ❌ 깔끔한 디자인 → ✅ Bento Grid + 8px 그리드 |
| 절제 | 과장하지 않는다. 결과와 근거로 말한다 | ❌ 혁신적인 교육 → ✅ Before/After 차이를 보세요 |
| 독립 | 조건을 달지 않는다. 남의 권위를 빌리지 않는다 | ❌ 전문가들이 추천하는 → ✅ 직접 해보면 압니다 |
| 본질 | 트렌드가 아니라 원리를 말한다 | ❌ 2026년 핫한 AI 도구 → ✅ 도구가 바뀌어도 남는 것 |

### 2.2 블랙리스트 (절대 쓰지 않는 표현)

**과장/클리셰:**
혁신적, 게임체인저, 미래를 선도하는, 패러다임 전환, 무한한 가능성, 꿈을 현실로

**공포 마케팅:**
뒤처지면 도태됩니다, 이거 모르면 손해, AI에 대체됩니다, 살아남으려면

**허위 약속:**
누구나 할 수 있다, 하루 만에 앱 런칭, 코딩 없이 개발자 되기

**기타:**
Ship (출시/배포/런칭/제품 완결로 대체), 깔딱 (사용 금지), 갓생 (사용 금지)

### 2.3 화이트리스트 (권장 표현)

**VDL다운 단어들:**
연구, 체계, 기준, 언어, 패턴, 의도, 설계, 명명, 주도, 완결, 직접, 정밀, 빌더

**결과를 묘사할 때:**
"의도대로 나왔다", "피드백을 기다리지 않는다", "내 판단으로 결과를 냈다", "넘기지 않고 끝까지 만들었다"

**변화를 묘사할 때:**
"Figma 밖으로 나왔다", "해석을 기다리지 않는다", "기준이 생기니 판단이 된다", "본업의 반경이 넓어졌다"

### 2.4 완급 조절: 합니다/해요 혼용

보이스 전체는 합니다체를 기본으로 하되, 독자와 감정을 공유하는 순간에만 해요체를 섞는다. 문단당 1~2회 이하.

| 체 | 역할 | 예시 |
|---|---|---|
| 합니다 | 서술, 선언, 논증 | "달라지지 않습니다", "순서가 반대입니다" |
| 해요 | 독자와 감정을 공유하는 순간 | "여전히 아쉬워요", "왜 어색한지 모르겠어요" |

해요체가 오는 위치: 페인포인트 묘사, 좌절 순간, 독자의 내면 독백에 가까운 문장.
해요체가 오지 않는 위치: VDL의 선언, 가치 제안, 결론, 미션.

### 2.5 채널별 톤 조절

보이스는 동일하되, 톤의 온도가 다르다:

| 채널 | 톤 | 문장 스타일 | 예시 |
|------|-----|-----------|------|
| Threads/X | 선언적, 캐주얼 | 1~3문장. 끊어 말하기. 반전 마무리 | "예쁘게 만들어줘"는 프롬프트가 아니다. 프롬프트는 기준이다. |
| 블로그/유튜브 | 실무적, 시연 중심 | Before/After 구조. 스크린샷·GIF 동반 | 같은 의도, 다른 프롬프트. 왼쪽은 3번 수정, 오른쪽은 1번에 끝 |
| 뉴스레터 | 솔직, 깊이 있음 | 에세이 + 솔루션 혼합. 개인 경험 허용 | 10년 동안 디자인하고 개발했다. 그래서 아는 게 있다 |
| 코호트/커뮤니티 | 동료적, 직접적 | 질문-답변, 리뷰, 피드백 | 이 프롬프트에서 빠진 건 "상태 전환" 명세다. 추가하면 달라진다 |
| B2B 제안 | 데이터 중심, 절제 | 수치, 사례, ROI | 코호트 1기 졸업생 중 68%가 3개월 내 직무 범위 확장을 보고했다 |

## 3. 콘텐츠 글 구조

### 3.1 기본 구조: 욕망 → 정답 → 다음 행동

```
제목 → 독자의 욕망을 건드린다 (스크롤 멈추기)
본문 → 정답을 준다 (가치 전달)
맺음 → 다음 행동을 유도한다 (팔로우, 저장, 클릭)
```

VDL은 "지시"보다 **"다음 행동 유도"**가 맞다. "팔로우하세요"보다 "Dictionary에서 직접 확인해보세요"가 VDL 톤이다.

### 3.2 고성과 글 패턴 × VDL 적용

**패턴 A: 반복 구문형** — 같은 구조 5회+ 반복, 마지막에 반전

```
디자이너가 코딩을 배우면, 프로토타입을 혼자 만듭니다.
디자이너가 코딩을 배우면, 개발자에게 설명할 필요가 없습니다.
디자이너가 코딩을 배우면, 아이디어가 나온 날 배포합니다.
디자이너가 코딩을 배우면, 사이드프로젝트가 수익이 됩니다.

단, 코딩을 "배우는" 게 아닙니다. AI에게 "시키는" 겁니다.
```

사용 맥락: Threads/X 숏폼. 최고 성과 패턴. 마지막 줄의 반전이 핵심.

**패턴 B: 선언/이분법형** — 100자 미만 선언. 동의 또는 반발.

```
"예쁘게 만들어줘"는 프롬프트가 아니다. 기준이 없으면 AI도 모른다.
```

```
도구를 잘 쓰는 사람과 도구로 잘 만드는 사람은 다르다.
```

```
코드를 몰라도 만들 수 있다는 말은 맞다. 잘 만들 수 있다는 말은 아니다.
```

사용 맥락: Threads/X. 글자 수 대비 반응 효율이 가장 높음. VDL 포지셔닝 대비 문구(2.5절)에서 소재 추출.

**패턴 C: 넘버링 리스트형** — 3~10개 항목. 제목에 숫자.

```
바이브 코딩으로 "개발자가 만든 티" 벗기는 5가지 디자인 키워드
```

```
프롬프트 한 줄 바꿨을 뿐인데 결과가 달라지는 패턴 7가지
```

사용 맥락: 저장/리포스트 유도. Dictionary 키워드 노출에 적합.

**패턴 D: 비유 서사형** — 일상 장면에서 시작, 디자인/바이브 코딩 원리로 전환.

```
카페에서 메뉴판을 보는데 "시그니처 블렌드"라고만 써 있다.
뭔지 모르겠다. 물어봐야 안다. — AI에게 "깔끔하게 만들어줘"라고 하는 거랑 같다.
"에티오피아 예가체프, 미디엄 로스트, 산미 강조"라고 쓰면 안 물어봐도 안다.
프롬프트도 같다. 정밀하면 한 번에 나온다.
```

사용 맥락: 블로그, 뉴스레터. 가장 깊은 인상. 도입 3줄 안에 몰입시켜야 한다.

### 3.3 제목 공식

```
[독자] + [과장 × (상황 + 숫자)] + [해결책/변화]
```

**VDL 제목 예시:**

| 구조 | 제목 |
|------|------|
| 독자 + 변화 | 디자이너가 개발자 없이 3일 만에 프로토타입 배포한 이야기 |
| 숫자 + 솔루션 | 프롬프트 3줄로 30시간 걸리던 작업을 끝냈습니다 |
| 상황 + 해결책 | "개발자가 만든 티" 벗기는 5가지 디자인 키워드 |
| 욕망 + 반전 | 바이브 코딩 강의 100개 들어도 안 되는 이유 — 빠진 건 코딩이 아니다 |
| 이분법 | 도구를 배우는 사람 vs 언어를 배우는 사람 |

**제목 블랙리스트:** "이거 모르면 손해", "안 보면 후회", "난리난", "충격적인"

### 3.4 프레이밍 원칙: 제품이 아니라 변화를 말한다

| ❌ 제품 프레이밍 | ✅ 변화 프레이밍 |
|----------------|----------------|
| Cursor 사용법 | 디자이너가 개발자 없이 프로토타입 배포하기 |
| AI 프롬프트 엔지니어링 팁 | 프롬프트 3줄로 30시간 걸리던 작업 끝내기 |
| 바이브 코딩 강의 | 디자인 의도를 끝까지 통제하는 법 |
| Vibe Dictionary 소개 | "예쁘게 만들어줘" 대신 쓸 수 있는 키워드 체계 |
| 디자인 시스템 이론 | 왜 8px인지 모르면 AI도 모른다 |

## 4. 채널 운영 컨벤션

### 4.1 콘텐츠 퍼널

```
Threads/X (숏폼, 1~3문장)     — 아이디어 검증. 반응 데이터 수집
    ↓ 반응 좋은 소재
블로그/유튜브 (미드폼, 시연)    — Before/After + 시각적 증거물 확장
    ↓ 심화 관심
뉴스레터/코호트 (롱폼, 체험)   — 에세이 + 솔루션. 팬 전환
```

숏폼 반응 데이터가 롱폼의 방향을 결정한다. 처음부터 길게 쓰지 않는다.

차별점: 텍스트 + 시각적 증거물(스크린샷, GIF, 데모 영상) 결합. 텍스트만으로 승부하지 않는다.

### 4.2 콘텐츠 유형 믹스

| 유형 | 비중 | 설명 |
|------|------|------|
| 가치 전달 (교육/인사이트) | 60% | Dictionary 키워드, Before/After, 채용 시장 분석 |
| 공감 (에세이) | 25% | 페인포인트 묘사, 실무자 일상, 파운더 경험 |
| 제품 언급 (코호트, Dictionary) | 10% | 직접 홍보는 전체의 10% 이하 |
| 커뮤니티 (질문, 토론) | 5% | 프롬프트 클리닉, 결과물 리뷰 |

순서: 유익하다 → 궁금하다 → 배우고 싶다. 이 순서를 거꾸로 하지 않는다.

### 4.3 재게시 전략

히트한 콘텐츠는 2~3개월 간격으로 재게시. 팔로워 대부분은 첫 게시를 보지 못했다.

재게시 규칙:
- 완전 복붙 금지. 항목 1개 추가하거나, 마지막 문장을 바꾸거나, 제목의 수식어를 조정
- 2회차가 1회차보다 반응이 높을 수 있다 (팔로워 성장 효과)
- 핵심 메시지(포지셔닝 대비 문구)는 반복할수록 각인된다

### 4.4 파운더(DDD) 보이스 규칙

| 구분 | DDD 개인 | VDL 브랜드 |
|------|---------|-----------|
| 톤 | 실무자의 솔직함 | 연구소의 정밀함 |
| 1인칭 | "나는", "10년 동안" | "VDL은", "우리 연구에서" |
| 경험 공유 | 허용 (개인 실패/성공) | 데이터와 사례로 |
| 의견 표현 | 허용 (단정적) | 근거 동반 |

게시글로 전문성, 댓글로 인성. 대부분은 게시글에만 신경 쓰지만, 유료 전환은 댓글에서 결정된다.

## 5. 용어 가이드

### 5.1 VDL 고유 용어

| 용어 | 정의 | 사용 맥락 |
|------|------|----------|
| 바이브 디자인 (Vibe Design) | 바이브 코딩 환경에서 디자인 의도를 구사하는 방법론 | 브랜드 정체성 설명 |
| Vibe Dictionary | AI가 이해하는 표준화된 디자인 어휘 체계 | 제품명, 리드 마그넷 |
| The Vibe Standard | Dictionary가 체현하는 표준 | 가치 제안 |
| System Over Drawing | 결과물이 아니라 기준을 먼저 설계한다 | 가치 제안, 교육 철학 |
| Design As Build | 설계가 곧 구현이다 | 가치 제안, 비전 |
| Design at the Speed of Thought | 생각의 속도로 만든다 | 슬로건 |

### 5.2 대체 용어 테이블

| ❌ 쓰지 않는 표현 | ✅ 대체 표현 | 이유 |
|-----------------|------------|------|
| Ship, 쉽하다 | 출시 / 배포 / 런칭 / 제품 완결 (맥락에 따라) | 브랜드 톤 불일치 |
| AI 코딩 | 바이브 코딩 | 개발자 범용 용어 → VDL 고유 맥락 |
| 수강생, 학생 | 빌더, 학습자 | 교육 업체 느낌 회피 |
| 강의, 수업 | 코스, 코호트 | 연구소 정체성. 코스는 콘텐츠 단위, 코호트는 운영 단위(기수) |
| 졸업 | 코호트 완료, 수료 | 학교 느낌 회피 |
| 선생님, 강사 | 프랙티셔너, 리드 | 현업에서 만들고 있는 사람이 알려준다는 톤 |

### 5.3 디자인 어휘 수준 조절

| 대상 | 어휘 수준 | 예시 |
|------|----------|------|
| 일반 대중 (SNS) | 일상어 + 핵심 키워드 1개 | "간격이 8px인 이유가 있다" |
| 관심층 (블로그) | 키워드 2~3개 + 시각적 설명 | "Bento Grid + semantic color token으로 프롬프트를 바꿨다" |
| 학습자 (코호트) | 전문 어휘 자유 사용 | "progressive disclosure 패턴의 상태 전환 로직" |

## 6. Before/After 포맷

VDL의 가장 강력한 증명 도구. 모든 채널에서 반복 사용.

### 6.1 표준 포맷

```
[Before]
프롬프트: "카드형 레이아웃, 둥근 모서리, 밝은 배경에 그림자 살짝"
결과: (스크린샷) — 나름 구체적으로 썼는데 매번 다르게 나옴

[After]  
프롬프트: "유저 상태 3종 → StatusBadge 컴포넌트 → 상태별 semantic color,
         카드 레이아웃 Bento Grid, 간격 8px 기준"
결과: (스크린샷) — 1번에 의도대로

차이를 만든 건 코딩 실력이 아니라 데이터, 컴포넌트, 스타일을 한 문장에 정의하는 언어다.
```

### 6.2 마무리 문구 풀

- "차이를 만든 건 코딩 실력이 아니라 디자인 어휘다."
- "같은 도구, 같은 AI. 다른 건 프롬프트에 담긴 기준뿐이다."
- "정밀하게 말하면 정밀하게 나온다. 그게 The Vibe Standard다."
- "코드를 바꾼 게 아니라 말을 바꿨다."
