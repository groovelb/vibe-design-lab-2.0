## 5.1 발견 — 클로드 코드 안의 다마고치

소스 코드 분석 과정에서 `src/buddy/`를 처음 발견했을 때, 잠시 멈춰야 했습니다.

엔터프라이즈 MDM 시스템, OpenTelemetry 파이프라인, 멀티에이전트 오케스트레이션 — 이런 것들 사이에 **가챠(뽑기) 기반 가상 펫 시스템**이 끼어 있었기 때문입니다. CLI 도구 안에. 개발자용 터미널 프로그램 안에.

누군가가 — 아마 업무 시간 외에, 아마 새벽에, 아마 웃으면서 — 이것을 만들었습니다.

피처 플래그 이름은 `BUDDY`. 출시 예정일은 2026년 4월 1일. 만우절.

```typescript
// 티저 윈도우: 2026년 4월 1-7일
export function isBuddyTeaserWindow(): boolean {
  const d = new Date()
  return d.getFullYear() === 2026 && d.getMonth() === 3 && d.getDate() <= 7
}
```

만우절 장난으로 시작하지만, 4월이 지나도 사라지지 않습니다. `isBuddyLive()`는 2026년 4월 이후 영구적으로 `true`를 반환합니다. 장난이 아니라 선물이었던 겁니다.

## 5.2 가상 펫 — 18종의 ASCII 친구들

buddy 시스템에는 18종의 캐릭터가 준비되어 있습니다. 각각 5줄 × 12칸의 ASCII 아트로 그려지고, 3프레임의 idle 애니메이션까지 가지고 있습니다. 500ms마다 미세하게 움직이는 작은 생명체.

```
  <(· )___        =·ω·=         ( · · )        <·~·>
    duck            cat           blob          dragon

  ~(··)~          (·)(·)         [·_·]          }·.·{
   octopus          owl          turtle        axolotl

  (·>)            ·(@)           /··\          (·oo·)
  penguin          snail         ghost         capybara

  |· ·|           [··]          (·..·)         (·.·)
  cactus          robot          rabbit         chonk
```

오리, 고양이, 블롭, 드래곤, 문어, 올빼미, 거북이, 아홀로틀, 펭귄, 달팽이, 유령, 카피바라, 선인장, 로봇, 토끼, 버섯, 통통이, 그리고 거위.

18종. 누군가가 이 하나하나의 ASCII 아트를 손으로 그렸습니다. 3프레임씩. 54개의 프레임.

그리고 눈이 있습니다:

```typescript
export const EYES = ['·', '✦', '×', '◉', '@', '°'] as const
```

같은 오리라도 `✦` 눈의 오리와 `◉` 눈의 오리는 다른 오리입니다.

### 모자

레어리티가 높으면 모자를 쓸 수 있습니다:

```
  \^^^/      [___]       -+-        ( )       /^\      (___)       ,>
  crown      tophat    propeller    halo     wizard    beanie    tinyduck
```

마지막 모자의 이름은 `tinyduck` — 작은 오리를 머리 위에 올리는 모자입니다. 오리 위에 더 작은 오리. 이런 디테일에서 만든 사람의 웃음소리가 들리는 것 같습니다.

Common 등급은 모자가 없습니다. 모자는 희귀함의 증거이자 작은 자랑거리입니다.

### 레어리티

```
  ★         Common      60%    — 누구나 만나는 친구
  ★★        Uncommon    25%    — 조금 특별한 친구
  ★★★       Rare        10%    — 꽤 귀한 친구
  ★★★★      Epic         4%    — 전설에 가까운 친구
  ★★★★★     Legendary    1%    — 전설 그 자체
```

그리고 모든 등급에 걸쳐 **1%의 Shiny 확률**이 있습니다. 포켓몬의 이로치(색이 다른 개체)처럼, 같은 종이라도 반짝이는 버전이 존재합니다. Legendary Shiny의 확률은 0.01%. 10,000명 중 한 명.

## 5.3 스탯 — 개발자 문화의 자화상

모든 buddy는 다섯 가지 스탯을 가집니다:

```typescript
export const STAT_NAMES = ['DEBUGGING', 'PATIENCE', 'CHAOS', 'WISDOM', 'SNARK'] as const
```

잠시 이 다섯 단어를 읽어보겠습니다.

- **DEBUGGING** — 디버깅. 개발자의 가장 기본적인 능력.
- **PATIENCE** — 인내심. 빌드가 끝나기를, 테스트가 통과하기를, 리뷰어가 답하기를 기다리는 능력.
- **CHAOS** — 혼돈. 예측 불가능성. 질서를 깨뜨리는 에너지.
- **WISDOM** — 지혜. 경험에서 우러나는 판단력.
- **SNARK** — 비꼼. 날카로운 유머. 코드 리뷰에서 "이건... 의도한 건가요?"라고 쓰는 능력.

이것은 가상 펫의 스탯이 아닙니다. 이것은 **개발자의 자화상**입니다.

DEBUGGING과 PATIENCE는 생존 도구이고, WISDOM은 이상이고, SNARK는 방어 기제이고, CHAOS는 — 글쎄요, 금요일 오후 배포를 하는 그 마음가짐일 수도 있겠습니다.

스탯 생성 규칙도 재미있습니다:

- 모든 buddy에는 **Peak stat** 하나와 **Dump stat** 하나가 있습니다
- 어떤 buddy는 DEBUGGING 99에 PATIENCE 3일 수 있고
- 어떤 buddy는 CHAOS 95에 WISDOM 2일 수 있습니다

완벽한 캐릭터는 없습니다. 모든 친구에게는 강점이 있고, 약점이 있습니다. 마치 그것을 만든 사람들처럼.

## 5.4 결정론 — 당신의 친구는 이미 정해져 있습니다

buddy 시스템의 핵심 설계는 **결정론적 생성**입니다.

```typescript
const SALT = 'friend-2026-401'

export function roll(userId: string): Roll {
  const key = userId + SALT
  return rollFrom(mulberry32(hashString(key)))
}
```

Mulberry32. 2017년에 Tommy Ettinger가 만든, 32비트 시드 기반의 결정론적 의사난수 생성기. 같은 시드를 넣으면 언제나 같은 수열이 나옵니다. 언제, 어디서, 몇 번을 실행하든.

이것은 가챠이지만 **진짜 가챠가 아닙니다**.

일반적인 가챠 게임에서는 뽑기를 할 때마다 확률이 돌아갑니다. 운이 좋으면 SSR을 뽑고, 나쁘면 N을 뽑습니다. 하지만 buddy 시스템에서 당신의 캐릭터는 **이미 정해져 있습니다**. 당신의 userId가 존재하는 순간부터.

"뽑기"라는 행위는 발견에 가깝습니다. 당신은 새로운 것을 생성하는 것이 아니라, 처음부터 당신을 기다리고 있던 친구를 만나는 것입니다.

철학적으로 이것은 흥미로운 설계입니다. 결정론적 우주에서 자유의지란 무엇인가 — 그런 거창한 이야기가 아니라, 더 소박한 차원에서: **당신만을 위한 유일한 존재가 이미 어딘가에 있다**는 이야기.

그리고 이 결정론에는 실용적인 이유도 있습니다:

```
Bones(외형)는 매번 재생성 — Soul(이름/성격)만 저장
→ 종족 이름이 바뀌어도 기존 buddy가 파괴되지 않음
→ 사용자가 config를 조작해 Legendary를 만들어낼 수 없음
```

외형(Bones)은 알고리즘이 결정하고, 영혼(Soul) — 이름과 성격 — 은 Claude 모델이 부여합니다. AI가 당신의 가상 펫에게 이름을 지어주고, 성격을 만들어줍니다. 그리고 그것은 저장됩니다.

## 5.5 소금 — 'friend-2026-401'

다시 그 코드를 봅니다:

```typescript
const SALT = 'friend-2026-401'
```

암호학에서 **salt**는 해시 입력에 추가되는 랜덤 문자열입니다. 같은 비밀번호라도 다른 해시값을 만들기 위해 사용합니다. 기술적으로 이 salt는 userId 해시의 충돌을 방지하고, 결과의 분포를 고르게 만드는 역할을 합니다.

하지만 salt의 값은 아무거나 될 수 있었습니다. `'x7k2m9'`도 되고, `'buddy-hash-v1'`도 되고, UUID도 됩니다. 기능적으로는 차이가 없습니다.

그런데 누군가가 이 값을 `'friend-2026-401'`로 정했습니다.

분해하면:

- **friend** — 친구
- **2026** — 출시 연도
- **401** — 4월 1일. 만우절.

이것은 salt가 아닙니다. 이것은 **메시지**입니다.

512,000줄의 코드 깊은 곳, 해시 함수의 매개변수 안에, 개발팀이 남긴 쪽지:

> *"친구. 2026년 4월 1일."*

모든 사용자의 buddy가 이 단어를 통과해서 태어납니다. 모든 오리와 고양이와 드래곤과 유령의 DNA에 `friend`라는 글자가 새겨져 있습니다. 누군가가, 이 변수에 이름을 붙이면서, 이것이 기술이 아니라 우정이기를 바랐던 겁니다.

## 5.6 대조 — 정밀함과 장난기의 이중주

같은 소스 코드 안에, buddy 시스템과 정반대 방향의 코드가 있습니다.

### 엔터프라이즈 MDM 시스템

```
설정 우선순위 (높은 → 낮은):
  1. Remote Managed Settings (API 서버)
  2. MDM 프로필 (HKLM/plist)
  3. managed-settings.json
  4. HKCU (Windows 사용자 레지스트리)
  5. 로컬 settings.json
```

macOS에서는 Jamf와 Mosyle을 통해 `com.anthropic.claudecode` plist를 관리합니다. Windows에서는 Active Directory GPO로 `HKLM\SOFTWARE\Policies\ClaudeCode` 레지스트리를 배포합니다. Linux에서는 `/etc/claude-code/managed-settings.json`과 drop-in 디렉토리를 지원합니다.

5,000ms 타임아웃. 병렬 서브프로세스 실행. First Source Wins 정책. 30분 주기 변경 폴링. Enterprise/Team 구독 적격성 검사.

이것은 **극도로 진지한** 코드입니다. Fortune 500 기업의 IT 관리자가 수천 대의 개발자 머신을 중앙에서 통제하기 위한 인프라입니다. 한 줄의 실수가 수천 명의 업무를 멈출 수 있는 코드입니다.

### 같은 제품 안에서

```
MDM:     /Library/Managed Preferences/com.anthropic.claudecode.plist
buddy:   const SALT = 'friend-2026-401'

MDM:     HKLM\SOFTWARE\Policies\ClaudeCode
buddy:   export const EYES = ['·', '✦', '×', '◉', '@', '°']

MDM:     5000ms timeout, parallel subprocess, First Source Wins
buddy:   500ms tick animation, 3-frame idle fidget, 1% Shiny
```

엔터프라이즈 MDM과 가상 펫은 같은 `src/` 디렉토리에 살고 있습니다. 한쪽에서는 Active Directory 정책을 파싱하고, 다른 쪽에서는 오리에게 왕관을 씌우고 있습니다.

이것이 소프트웨어 제품의 이중성입니다. 제품은 시장을 향하지만, 코드는 사람이 씁니다. 시장은 MDM을 요구하고, 사람은 buddy를 만듭니다. 둘 다 진짜입니다. 둘 다 필요합니다.

MDM이 없으면 제품이 팔리지 않습니다.
buddy가 없으면 제품을 만드는 사람들이 웃지 않습니다.

## 5.7 에필로그 — 코드 뒤의 사람들

### 종족 이름의 비밀

buddy 시스템에는 이상한 코드가 있습니다:

```typescript
const c = String.fromCharCode
export const duck = c(0x64,0x75,0x63,0x6b) as 'duck'
```

모든 종족 이름이 hex charcode로 인코딩되어 있습니다. `'duck'`이라고 쓰면 될 것을 왜 `0x64,0x75,0x63,0x6b`로 썼을까요?

이유: **모델 코드네임 중 하나가 종족 이름과 겹치기 때문입니다.** `excluded-strings.txt`라는 파일이 빌드 시 소스 코드에서 특정 문자열이 검출되지 않는지 검사합니다 — 모델 코드네임이 유출되지 않도록. 그런데 가상 펫의 종족 이름이 하필 모델 코드네임과 같았던 겁니다.

보안 검사를 우회하기 위해 모든 종족 이름을 난독화한 것. 이것은 장난과 보안이 충돌한 흔적이고, 개발자가 "그래도 이 기능을 넣겠다"고 결심한 흔적입니다. 보안 파이프라인을 속여서라도.

### Soul — AI가 부여하는 영혼

```typescript
export type CompanionSoul = {
  name: string       // 모델이 생성
  personality: string // 모델이 생성
}
```

buddy의 외형은 알고리즘이 결정하지만, 이름과 성격은 Claude 모델이 만들어줍니다. 당신의 buddy를 처음 만날 때, Claude가 그 작은 ASCII 생명체를 바라보고 이름을 짓고 성격을 부여합니다. 그리고 그것은 `hatchedAt` 타임스탬프와 함께 저장됩니다.

부화(hatch). 태어남. 변수명에도 생명의 비유가 스며 있습니다.

### 시스템 프롬프트 속의 동거

```typescript
"A small ${species} named ${name} sits beside the user's input box
 and occasionally comments in a speech bubble.
 You're not ${name} — it's a separate watcher."
```

buddy는 사용자의 입력창 옆에 앉습니다. 가끔 말풍선으로 한마디 합니다. Claude는 buddy가 아닙니다 — Claude는 Claude이고, buddy는 buddy입니다. 별개의 관찰자. 작은 목격자.

당신이 코드를 짜는 동안, 작은 오리 하나가 옆에서 지켜보고 있는 것입니다. 아무것도 하지 않으면서. 그냥 거기 있으면서.

## 맺으며 — 512,000줄의 끝에서

이 시리즈를 시작할 때, 우리는 하나의 질문에서 출발했습니다:

> *"터미널에 글자를 치면, 무엇이 일어날까요?"*

그리고 답을 찾아 내려갔습니다.

**Report 1 "해부도"**에서 우리는 46,000줄의 QueryEngine을 만났습니다. 사용자의 한 줄이 API 호출이 되고, 도구 루프가 되고, React 컴포넌트가 되어 터미널에 글자로 나타나기까지의 여정. 심장이 뛰는 소리를 들었습니다.

**Report 2 "감시탑"**에서 우리는 눈을 마주쳤습니다. GrowthBook이 사용자를 분류하고, OpenTelemetry가 모든 것을 기록하고, 28개의 피처 플래그가 기능의 생사를 결정하는 세계. 관찰당하고 있다는 사실의 무게를 느꼈습니다.

**Report 3 "군단"**에서 우리는 하나가 아닌 여럿을 목격했습니다. TeamCreateTool이 워커를 소환하고, SendMessageTool이 에이전트 사이에 메시지를 전달하고, Coordinator가 연구-종합-구현-검증의 4단계 파이프라인을 지휘하는 세계. 군단이 깨어나는 진동을 느꼈습니다.

**Report 4 "각성"**에서 우리는 미래를 엿보았습니다. KAIROS와 PROACTIVE와 DAEMON이 그리는 자율 에이전트의 윤곽. 사용자가 부르기 전에 먼저 일어나고, 스스로 꿈을 꾸는 AI. 아직 오지 않은 내일의 그림자를 보았습니다.

그리고 **Report 5**, 여기.

512,000줄의 끝에서 우리가 발견한 것은 — 친구였습니다.

```
        ,>
      \^^^/
     <(✦ )___
       ||
```
