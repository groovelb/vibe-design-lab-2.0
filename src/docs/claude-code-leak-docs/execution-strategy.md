## 1. 도구/에이전트 역량 인벤토리

### 1.1 직접 도구 (Main Context에서 실행)

| 도구 | 강점 | 약점 | 적합 작업 |
|------|------|------|-----------|
| **Read** | 즉시 실행, 정확한 라인 지정, 이미지/PDF 지원 | 1회 2000줄 제한, 대용량 파일 다회 호출 필요 | M1 소규모 파일 정독 |
| **Grep** | ripgrep 기반 빠른 전체 검색, 정규식 지원, 컨텍스트 출력 | 결과량 많으면 head_limit 필요, 의미 분석 불가 | M2 패턴 마이닝 |
| **Glob** | 빠른 파일 패턴 매칭, 수정 시간 정렬 | 내용 검색 불가 | 파일 목록 확인 |
| **Bash** | 자유도 높음, wc/sort/pipe 조합 | 전용 도구보다 UX 불리 | 통계, 파일 크기 확인 |
| **Write** | 결과 영속화 | 기존 파일 덮어씀 | 산출물 저장 |

### 1.2 서브에이전트 (Agent Tool)

| 에이전트 타입 | 강점 | 약점 | 적합 작업 |
|--------------|------|------|-----------|
| **Explore** | 코드베이스 탐색 특화, 다수 파일 순회, 독립 컨텍스트 | 코드 수정 불가, 결과 요약만 반환, 내부 과정 불투명 | M1(대규모 파일), M2, M3, M4 |
| **general-purpose** | 모든 도구 사용 가능, 복잡한 다단계 추론 | 느림, 컨텍스트 비용 높음 | M4+M5+M6 복합 분석 |
| **Plan** | 아키텍처 설계 특화, 구조적 사고 | 파일 수정 불가, 탐색은 Explore보다 느림 | 의존성 분석, 관계 설계 |

### 1.3 병렬 실행 역량

- **동시 에이전트**: 단일 메시지에서 **최대 3-4개** 병렬 호출 가능 (실무적 최적치)
- **백그라운드 에이전트**: `run_in_background: true`로 비동기 실행, 완료 시 알림
- **직접 도구 병렬**: 단일 메시지에서 다수 Read/Grep/Glob 동시 호출 가능

## 2. 제약 조건 분석

### 2.1 컨텍스트 윈도우 제약

| 제약 | 영향 | 대응 |
|------|------|------|
| 대화가 길어지면 과거 메시지 자동 압축 | 이전 분석 결과 소실 가능 | **즉시 파일 저장** — 분석 완료 즉시 docs/에 Write |
| 에이전트 결과는 메인 컨텍스트에 텍스트로 반환 | 대용량 결과가 컨텍스트 소비 | 에이전트에 **요약 지시** 명시 |
| 압축 후 이전 에이전트 ID 유실 가능 | resume 불가 | Phase별로 완결, 크로스 Phase 의존 최소화 |

### 2.2 파일 크기 제약

| 대상 파일 | 크기 | Read 횟수 (2000줄 기준) | 전략 |
|-----------|------|------------------------|------|
| QueryEngine.ts | ~46K줄 | ~23회 | Grep으로 핵심 함수 위치 파악 → 타겟 Read |
| prompts.ts | ~54KB (~1,500줄+) | 1-2회 | Explore 에이전트에 전체 위임 |
| REPL.tsx | ~4,600줄 | ~3회 | 상위 200줄 + Grep 타겟 |
| coordinatorMode.ts | 중간 | 1회 | 직접 Read |
| swarm 백엔드 9파일 | 각 100-400줄 | 파일당 1회 | Explore 에이전트에 일괄 위임 |

### 2.3 존재하지 않는 코드

| 대상 | 상태 | 대응 |
|------|------|------|
| `src/proactive/` | 디렉토리 미존재, feature-gated require() | Grep으로 참조 역추적만 가능 |
| Dream 스킬 원본 | `require('./dream.js')` — 컴파일 아티팩트? | `skills/bundled/` 하위에서 dream 관련 파일 탐색 |
| KAIROS 하위 도구 | tools 디렉토리에 존재 확인 필요 | Glob으로 존재 여부 먼저 확인 |

## 3. 22개 작업별 적합성 평가

### 범례

- **실행 방식**: `Direct` (직접 도구) / `Explore` / `General` / `Hybrid` (직접+에이전트 혼합)
- **병렬 그룹**: 같은 그룹은 동시 실행 가능
- **난이도**: ⬜ 쉬움 / 🟨 보통 / 🟥 어려움
- **신뢰도**: 예상 결과 품질 (높/중/낮)

### Phase 1: 핵심 엔진 + 핵심 소재

#### R1-B: 시스템 프롬프트 조립

| 항목 | 평가 |
|------|------|
| **실행 방식** | `Explore` (very thorough) |
| **이유** | prompts.ts 54KB + systemPromptSections.ts + systemPrompt.ts 3파일 교차 분석 필요. 단일 Read로는 컨텍스트 부족 |
| **에이전트 프롬프트 핵심** | "prompts.ts의 전체 구조를 파악하라. (1) 어떤 섹션들로 구성되는가, (2) 각 섹션의 내용 요약, (3) 동적으로 삽입되는 컨텍스트 종류, (4) 캐싱 전략. 프롬프트 원문 중 핵심 지시문을 인용하라" |
| **보조 도구** | Grep(`systemPromptSection\(`, `DANGEROUS_uncached`) → 섹션 등록 전체 목록 사전 확보 |
| **난이도** | 🟥 어려움 (54KB, 복잡한 조건부 조립) |
| **신뢰도** | 높음 — 텍스트 기반 분석이라 에이전트 적합 |
| **병렬 그룹** | P1-A |
| **리스크** | 프롬프트가 너무 길어 에이전트 컨텍스트 내에서 잘릴 수 있음 |
| **대응** | 섹션별로 분리 요청, "구조 먼저 → 핵심 섹션 원문 인용" 2단계 |

### 전체 실행 가능성

| 평가 항목 | 판정 | 근거 |
|-----------|------|------|
| **기술적 실현 가능성** | ✅ 가능 | 모든 작업이 Read/Grep/Explore 조합으로 수행 가능 |
| **품질 신뢰도** | ⚠️ 중상 | 22개 중 19개 높음, 3개 중간-낮음 (R1-A, R4-C, R5-B) |

| **병렬화 효율** | 높음 | 4개 Phase, Phase 당 2-3 에이전트 동시 실행 |

### 핵심 권고

1. **QueryEngine(R1-A)은 "완전 이해"를 포기하고 "핵심 루프 추출"에 집중** — 46K줄 전체 분석은 비현실적. 실행 루프 + 도구 호출 + 스트리밍 3가지만 확보
2. **proactive(R4-C)은 "추론 기반"임을 명시** — 소스 미존재, 참조 역추적만 가능. 확정적 서술 금지
3. **Phase 완료마다 반드시 docs/research/ 저장** — 컨텍스트 압축에 대한 유일한 방어선
4. **에이전트 프롬프트에 산출물 포맷 강제** — "노드/엣지 제안" 요구를 빠뜨리면 후속 가공 비용 급증
